# 리스크와 우선순위

## 1. 핵심 리스크 요약

### P0. 현재 사용자 경험과 직접 충돌하는 문제

- `2026-03-25` 기준 모집 날짜와 D-day가 이미 낡아 있다.
- member 영역에 사실상 인증과 권한이 없다.
- 일부 내부 링크가 현재 라우트와 어긋난다.
- public과 member 경계가 흐린 링크가 존재한다.

### P1. 구조 확장을 막는 문제

- 데이터가 페이지마다 하드코딩되어 중복과 드리프트가 크다.
- 페이지별 중복 UI가 많아 수정 비용이 크다.
- 비활성 페이지가 함께 남아 있어 정보 구조가 흐려진다.
- 디자인 토큰 체계가 이원화되어 있다.

### P2. 협업과 운영 리스크

- 문서와 가이드라인이 비어 있다.
- 테스트, CI, lint, typecheck가 없다.
- 배포 설정이 없다.
- 의존성이 과하게 넓다.

## 2. 가장 먼저 고쳐야 할 것

### 1단계: 사실관계와 경계 정리

- public/member 링크 정리
- 죽은 라우트와 고아 페이지 분리
- 낡은 날짜와 상태값 정리
- README와 분석 문서 연결

이 단계는 구현 난도보다 "잘못된 상태를 보여주는 문제"를 빠르게 줄이는 데 목적이 있다.

### 2단계: 운영 가능한 기반 만들기

- 인증과 권한 모델 확정
- 공통 도메인 데이터 구조 정의
- 폼 제출, 예약, 다운로드 같은 핵심 액션의 실제 처리 방향 결정
- 공통 페이지 헤더와 카드 패턴 추출

### 3단계: 레포 품질 올리기

- lint, typecheck, build 스크립트 정비
- 배포 타깃과 rewrite 규칙 정리
- `.gitignore`와 package policy 정리
- 문서와 가이드라인 갱신

## 3. 권장 작업 순서

1. 라우트와 정보 구조 정리
2. public 공통 shell 정리
3. member 인증/권한 설계
4. 더미 데이터 중앙화
5. 큰 페이지 분해
6. 운영 도구와 품질 게이트 도입

## 5. 실무적으로 가장 중요한 판단

이 프로젝트는 지금 당장 "예쁘게 다듬는 작업"보다 "무엇이 실제 기능이고 무엇이 시안인지 분리하는 작업"이 더 중요하다.

그 이유

- 사용자는 목업을 실제 기능으로 오해할 수 있다.
- 개발자는 구현 우선순위를 잡기 어렵다.
- 문서와 코드가 어긋난 상태에서는 이후 작업 속도가 급격히 떨어진다.
