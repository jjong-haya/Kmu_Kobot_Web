# KOBOT Web 작업 정리본

## 1. 이번에 바로잡은 핵심

`login_id`는 단순 입력칸이 아니라 로그인에 쓰이는 계정 식별자입니다.

그래서 아래 조건이 반드시 필요합니다.

- 전체 회원 기준 중복 불가
- 저장 전 소문자 정규화
- 최초 설정 후 일반 사용자는 직접 변경 불가
- 로그인하지 않은 사용자가 ID 존재 여부를 마음대로 조회할 수 없게 제한
- 동시에 같은 ID를 저장하려 해도 DB에서 최종 차단

## 2. 반영한 내용

| 구분 | 내용 |
| --- | --- |
| DB | `is_login_id_available` RPC 추가 |
| DB | 기존 `profiles_login_id_unique_idx`로 최종 중복 차단 유지 |
| 프론트 | ID 입력 후 blur 시 중복 확인 |
| 프론트 | 회원가입 요청 시 저장 직전 다시 중복 확인 |
| UX | 중복이면 상단 알림이 아니라 ID 입력칸 아래에 표시 |
| UX | 제출 시 중복이면 ID 입력칸으로 스크롤하고 빨간 테두리/진동 표시 |
| 문서 | 작업용 문서는 영어로 정리, 사용자 정리본은 이 문서처럼 한국어로 분리 |

## 3. 왜 이렇게 해야 하는가

ID 로그인은 내부적으로 다음 흐름입니다.

```text
login_id -> email 찾기 -> Supabase password 로그인
```

따라서 같은 ID가 두 계정에 존재하면 어떤 계정으로 로그인해야 하는지 모호해집니다.

UI에서만 중복을 막으면 동시에 제출하거나 브라우저를 우회하는 경우를 막을 수 없습니다. 그래서 DB 제약과 RPC 검증이 함께 필요합니다.

## 7. DDD 스킬을 이렇게 바꾼 이유

DDD는 “문서를 많이 쓰는 방식”이 아니라, 실제 동아리 운영 규칙을 코드로 옮기기 전에 용어, 권한, 상태, 예외, 보안 규칙을 끝까지 확인하는 방식입니다.

이번에 스킬과 문서에 박아둔 핵심은 다음입니다.

- 구현 전에 먼저 현재 상태, 의미, 왜 중요한지, 진행 방식을 설명합니다.
- 도메인 이해부터 검증까지 12단계를 순서대로 진행합니다.
- 검증 단계에서 사소한 질문까지 전부 목록화합니다.
- 질문이 하나라도 남으면 다시 1단계로 돌아갑니다.
- 질문을 해결하면서 생긴 코드나 문서 변경도 다시 질문을 만들 수 있으므로 반복해서 확인합니다.
- 루프를 닫기 전에는 최소 3개 관점, 즉 도메인, 구현, 리스크 검토가 필요합니다.
- 의견이 갈리면 무시하지 않고 `accepted`, `rejected`, `deferred`, `needs-rework` 중 하나로 기록합니다.

즉, 앞으로는 “대충 맞겠지”로 넘어가지 않고, 왜 이 규칙이 필요한지와 어떤 데이터/권한/화면에 영향을 주는지 확인한 다음 구현합니다.

## 8. 문서 이름의 의미

`13-domain-risk-review-2026-05-01.md`는 기능 명세서가 아니라 현재 프로젝트에서 위험한 도메인 규칙을 따로 잡아둔 작업 문서입니다.

예를 들면 `login_id`, 닉네임, 초대 코드, 권한, 투표 익명성, 연락 요청처럼 단순 입력칸처럼 보여도 실제로는 중복, 권한 상승, 개인정보 노출, 감사 로그 위조 같은 문제가 생길 수 있는 요소를 모아둔 문서입니다.

작업 문서는 AI와 개발자가 정확히 이어서 작업하기 좋게 영어로 유지하고, 사람이 빠르게 이해해야 하는 정리본은 이 파일처럼 한국어로 분리합니다.

## 9. 이번 DDD 재시작에서 정리한 구조

프로젝트 전체를 다시 1단계부터 훑으면서 문서를 두 종류로 나눴습니다.

첫 번째는 계속 누적되는 원장입니다.

- `02-domain-discovery.md`: 도메인 트리, 용어 사전, Aggregate, 불변조건
- `03-event-storming.md`: Actor, Command, Event, Policy, Read Model
- `04-data-schema-and-security.md`: 현재 DB 구조, RLS/RPC, 보안 경계, 빠진 제약
- `05-functional-spec.md`: 기능별 요구사항과 연결된 질문
- `06-design-spec.md`: 화면별 도메인, UX 상태, 금지 문구
- `07-implementation-plan.md`: 어떤 순서로 구현할지

두 번째는 매번 작업할 때 처리하는 검증 질문 원장입니다.

- `14-verification-question-ledger.md`

여기에는 현재 해결되지 않은 질문을 `active` 상태로 쌓아둡니다. 질문이 해결되면 그냥 삭제하지 않고, 해결 기록으로 옮깁니다.

즉 앞으로는 이런 흐름입니다.

```text
도메인 트리에서 작업할 가지 선택
-> 검증 질문 원장에서 관련 질문 확인
-> 질문 해결 또는 작업으로 전환
-> 해결하면서 새 질문이 생기면 다시 원장에 추가
-> 해당 도메인 1단계부터 다시 검증
```

## 10. 현재 가장 중요한 P0 질문

아직 기능 구현 전에 먼저 해결해야 하는 핵심 질문은 다음입니다.

| ID | 질문 요약 | 왜 중요한가 |
| --- | --- | --- |
| Q-AUTH-003 | Supabase Auth Hook이 실제로 연결되어 있는가 | SQL 함수만 있고 Dashboard 연결이 없으면 학교 계정 제한이 실제로 강제되지 않습니다. |
| Q-AUTH-004 | ID 로그인은 active 계정만 허용할 것인가 | pending, suspended, alumni가 ID 로그인으로 들어올 수 있는지 결정해야 합니다. |
| Q-AUTHZ-001 | maintainer를 project_operator로 바꿀 것인가 | 현재 helper가 maintainer/delegate를 팀장처럼 볼 위험이 있습니다. |
| Q-PROJECT-001 | 공식 팀장이 비공개 프로젝트 내부 자료를 어디까지 볼 수 있는가 | 비공개 프로젝트 자료 노출과 승인 검토 권한이 충돌합니다. |
| Q-INVITE-001 | 초대 코드 max_uses와 used 상태를 어떻게 처리할 것인가 | 여러 명이 쓰는 코드와 1회용 코드의 상태 모델이 충돌할 수 있습니다. |
| Q-VOTE-001 | 익명 투표의 익명성 수준은 어디까지인가 | 화면에서만 익명인지, 운영진도 개별 선택을 못 보는지 명확해야 합니다. |
| Q-AUDIT-001 | 일반 사용자가 감사 로그를 직접 insert할 수 있는가 | 감사 로그 위조 위험이 있습니다. |
| Q-AUDIT-002 | 감사 로그에 민감정보 원문을 저장할 수 있는가 | 전화번호, 학번, 연락처, 투표 선택값, 비공개 README 원문 보관 문제가 생깁니다. |
