# KOBOT Web 작업 정리본

## 1. 이번에 바로잡은 핵심

`login_id`는 단순 입력칸이 아니라 로그인에 쓰이는 계정 식별자입니다.

그래서 아래 조건이 반드시 필요합니다.

- 전체 회원 기준 중복 불가
- 저장 전 소문자 정규화
- 최초 설정 후 일반 사용자는 직접 변경 불가
- 로그인하지 않은 사용자가 ID 존재 여부를 마음대로 조회할 수 없게 제한
- 동시에 같은 ID를 저장하려 해도 DB에서 최종 차단

## 2. 반영한 내용

| 구분 | 내용 |
| --- | --- |
| DB | `is_login_id_available` RPC 추가 |
| DB | 기존 `profiles_login_id_unique_idx`로 최종 중복 차단 유지 |
| 프론트 | ID 입력 후 blur 시 중복 확인 |
| 프론트 | 회원가입 요청 시 저장 직전 다시 중복 확인 |
| UX | 중복이면 상단 알림이 아니라 ID 입력칸 아래에 표시 |
| UX | 제출 시 중복이면 ID 입력칸으로 스크롤하고 빨간 테두리/진동 표시 |
| 문서 | 작업용 문서는 영어로 정리, 사용자 정리본은 이 문서처럼 한국어로 분리 |

## 3. 왜 이렇게 해야 하는가

ID 로그인은 내부적으로 다음 흐름입니다.

```text
login_id -> email 찾기 -> Supabase password 로그인
```

따라서 같은 ID가 두 계정에 존재하면 어떤 계정으로 로그인해야 하는지 모호해집니다.

UI에서만 중복을 막으면 동시에 제출하거나 브라우저를 우회하는 경우를 막을 수 없습니다. 그래서 DB 제약과 RPC 검증이 함께 필요합니다.

## 4. 문서 구조 정리

앞으로 문서는 이렇게 나눕니다.

| 문서 종류 | 언어 | 목적 |
| --- | --- | --- |
| 작업용 스펙, 체크리스트, 리뷰 로그 | 영어 | AI와 개발자가 정확하게 이어서 작업하기 위함 |
| 사용자 정리본 | 한국어 | 현재 무엇이 정해졌고 무엇이 남았는지 빠르게 확인하기 위함 |

이전의 `DDD 재검증 문서`라는 표현은 부정확해서 제거했습니다.

대신 작업용 문서는 `13-domain-risk-review-2026-05-01.md`로 바꿨습니다.

의미는 “현재 도메인에서 위험한 불변조건과 다음 P0 작업을 점검하는 내부 작업 문서”입니다.

## 5. 아직 남은 중요한 작업

| 우선순위 | 작업 | 이유 |
| --- | --- | --- |
| P0 | 권한을 scope/source 기반 capability로 분리 | 프로젝트 팀장이 운영진 권한처럼 보이는 문제 방지 |
| P0 | 감사 로그 직접 insert 제한 | 로그 위조 방지 |
| P0 | 초대 코드 사용 RPC 구현 | 만료/동시 사용/재사용 문제 방지 |
| P0 | 투표 익명성 수준 명확화 | 사용자가 기대하는 익명성과 실제 DB 구조 불일치 방지 |

## 7. DDD 스킬을 이렇게 바꾼 이유

DDD는 “문서를 많이 쓰는 방식”이 아니라, 실제 동아리 운영 규칙을 코드로 옮기기 전에 용어, 권한, 상태, 예외, 보안 규칙을 끝까지 확인하는 방식입니다.

이번에 스킬과 문서에 박아둔 핵심은 다음입니다.

- 구현 전에 먼저 현재 상태, 의미, 왜 중요한지, 진행 방식을 설명합니다.
- 도메인 이해부터 검증까지 12단계를 순서대로 진행합니다.
- 검증 단계에서 사소한 질문까지 전부 목록화합니다.
- 질문이 하나라도 남으면 다시 1단계로 돌아갑니다.
- 질문을 해결하면서 생긴 코드나 문서 변경도 다시 질문을 만들 수 있으므로 반복해서 확인합니다.
- 루프를 닫기 전에는 최소 3개 관점, 즉 도메인, 구현, 리스크 검토가 필요합니다.
- 의견이 갈리면 무시하지 않고 `accepted`, `rejected`, `deferred`, `needs-rework` 중 하나로 기록합니다.

즉, 앞으로는 “대충 맞겠지”로 넘어가지 않고, 왜 이 규칙이 필요한지와 어떤 데이터/권한/화면에 영향을 주는지 확인한 다음 구현합니다.

## 8. 문서 이름의 의미

`13-domain-risk-review-2026-05-01.md`는 기능 명세서가 아니라 현재 프로젝트에서 위험한 도메인 규칙을 따로 잡아둔 작업 문서입니다.

예를 들면 `login_id`, 닉네임, 초대 코드, 권한, 투표 익명성, 연락 요청처럼 단순 입력칸처럼 보여도 실제로는 중복, 권한 상승, 개인정보 노출, 감사 로그 위조 같은 문제가 생길 수 있는 요소를 모아둔 문서입니다.

작업 문서는 AI와 개발자가 정확히 이어서 작업하기 좋게 영어로 유지하고, 사람이 빠르게 이해해야 하는 정리본은 이 파일처럼 한국어로 분리합니다.
