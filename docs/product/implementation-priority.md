# 구현 우선순위

이 문서는 Kobot 멤버 워크스페이스 정책을 실제 개발 작업으로 나누기 위한 우선순위 문서다.

기준 문서는 다음과 같다.

- [출시 전 법적 고지 및 개인정보 체크리스트](./legal-and-compliance-checklist.md)

공식 웹 도메인은 `kobot.kookmin.ac.kr`을 기준으로 한다.

운영 문의와 개인정보 문의 공식 메일은 `kobot@kookmin.ac.kr`을 기준으로 한다.

Google OAuth 동의 화면 서비스명은 `KOBOT`으로 둔다.

Google OAuth 사용자 유형은 `External`을 기준으로 한다.

Google OAuth Web Client ID는 `90336747619-5vb0mtujtvai7md1mufaebc9bpjp11gc.apps.googleusercontent.com`이다.

2026-04-28 확인 기준 `https://kobot.kookmin.ac.kr/` 루트는 Vercel에서 정상 응답한다.

현재 배포 HTML 제목은 `Kobot 웹사이트 제작`이다.

React Router 하위 경로가 직접 접속 시 404가 되지 않도록 Vercel SPA rewrite 설정을 유지해야 한다.

## 1. 우선순위 원칙

1차 구현은 실제 동아리원이 로그인하고, 승인받고, 기본 운영을 시작할 수 있는 최소 기능을 목표로 한다.

우선순위 판단 기준은 다음과 같다.

- 로그인과 회원 식별이 먼저다.
- 개인정보 고지와 동의 흐름은 기능보다 먼저 맞춰야 한다.
- 권한과 감사 기록은 나중에 붙이는 기능이 아니라 처음부터 같이 설계한다.
- 공개 페이지보다 멤버 워크스페이스 운영 기능을 먼저 안정화한다.
- 연락, 투표, GitHub 연동처럼 범위가 커질 수 있는 기능은 1차 범위를 작게 자른다.

## 2. 1차 구현 범위

### 2.1 인증과 진입 의도 보존

1차에서 반드시 구현한다.

- Google 최초 로그인
- Supabase Authentication Google Provider 활성화
- Google OAuth 사용자 유형 `External`
- `kookmin.ac.kr` 학교 메일 확인
- 학교 메일이 아닌 계정 안내 페이지
- 학교 메일이 아닌 계정의 탈퇴 및 재가입 요청
- 로그인 후 아이디 생성 안내 팝업
- 아이디/비밀번호 로그인
- Google 로그인 유지
- 다시 보지 않기 처리
- 공유 링크 클릭 후 로그인해도 원래 흐름으로 돌아오는 진입 의도 보존

왜 먼저인가?

로그인과 진입 의도 보존이 없으면 초대 링크, 프로젝트 참여, 연락 요청, 투표 알림이 전부 자연스럽게 이어지지 않는다.

### 2.2 프로필 필수 설정

1차에서 반드시 구현한다.

- 닉네임 필수
- 실명 필수
- 학번 필수
- 전화번호 필수
- 단과대 입력
- 학과 입력
- 소속 동아리 입력
- 공개 페이지 개인 표시 설정
- 이메일 알림 설정

닉네임 정책은 다음을 1차부터 적용한다.

- 현재 활동 사용자 기준 중복 불가
- 대소문자 구분 없이 중복 검사
- 탈퇴자 닉네임과 이전 닉네임 재사용 가능
- 변경은 7일에 1번
- 길이 2자 이상 12자 이하
- 한글, 영문, 숫자, 공백 허용
- 사용자가 직접 `_` 입력 불가
- 저장 시 공백은 `_`로 변환
- 화면 표시 시 `_`는 공백처럼 표시
- 금칙어와 부적절 닉네임 검사
- 변경 이력 최근 5개 표시
- 이전 닉네임 비공개 선택 가능

왜 먼저인가?

내부 페이지에서는 익명 표시 없이 닉네임이 기본 식별자로 쓰이고, 프로젝트 참여자 목록에서는 실명과 연락처가 필요하다.

### 2.3 사용자 상태와 권한

- 승인 대기
- 활동 부원

- 졸업

- 프로젝트 팀원

- IoT팀
- 연구팀

### 2.4 개인정보 고지와 문서 링크

- 이용약관 또는 회원 운영정책 페이지

- 운영자 정보와 문의처
- 로그인 페이지 하단 문서 링크
- 공개 페이지 푸터 문서 링크

문서 URL은 다음을 기준으로 한다.

왜 먼저인가?

실명, 학번, 전화번호, 연락 요청, 투표, 감사 기록을 다루므로 고지 문서는 실제 배포 전에 먼저 있어야 한다.

- 대상 공식 팀
- 행위 유형

- 변경 값
- 사유 또는 메모
- 생성 시각
- 승인자
- 승인 시각

- 기본 1년 보관
- 1년 후 삭제

1차에서 구현한다.

- 중요 알림 기본 켜짐
- 일반 알림 선택형
- 중요 이메일 알림도 끌 수 있음
- 끌 때 경고 문구 표시

왜 먼저인가?

승인, 투표, 연락 요청, 권한 이전은 알림이 없으면 흐름이 끊긴다.

### 2.8 연락 요청

1차에서 구현한다.

- 공개형 프로필의 `연락하기`
- 실시간 채팅이 아닌 연락 요청
- 연락 이유 입력
- 연락 이유 최소 글자 수 제한 없음
- 요청자 연락처 최소 1개 첨부
- 연락처 종류: 전화번호, 이메일, 카카오톡 ID, 디스코드 ID, GitHub, 기타
- 받은 요청
- 보낸 요청
- 상태: 대기, 수락, 거절
- 수락 시 수락자가 공개할 연락처 최소 1개 선택
- 수락 후 연락처 공개 취소 불가
- 거절 사유 선택 입력
- 입력된 거절 사유는 요청자에게 표시
- 대기 3일 후 자동 거절
- 자동 거절 사유 `미응답`
- 자동 거절 후 24시간 뒤 재요청 가능
- 명시적 거절 후 7일 뒤 재요청 가능
- 이미 수락된 요청이 있어도 재요청 가능
- 원본 1년 보관 후 삭제
- 각자 자기 화면에서 삭제 가능

남용 방지는 다음을 1차부터 적용한다.

- 기본 일일 제한 없음
- 스팸 신고 가능
- 회장/부회장 신고 확인과 조치 가능
- 10분 안에 3건 이상 경고 팝업
- 10분 안에 6건 이상 10분 쿨다운
- 1시간 안에 15건 이상 운영진 검토 큐
- 동일/유사 내용 반복 감지
- 봇성 패턴 즉시 제한

왜 1차인가?

공개형 프로필과 프로젝트 협업을 연결하는 핵심 기능이지만, 실시간 채팅보다 훨씬 작게 구현할 수 있다.

1차에서 구현한다.

- 공식 팀 기반 프로젝트
- 개인 또는 자율 프로젝트
- 프로젝트 승인/반려
- 승인 전 공유 링크
- 승인된 프로젝트만 목록 노출

- 프로젝트 참여 신청
- 프로젝트 팀원 승인/반려
- 프로젝트 팀장
- 프로젝트 팀원
- 프로젝트 참여자 목록

표시 항목은 다음과 같다.

- 학과

왜 1차인가?

사용자가 실제로 쓰는 멤버 워크스페이스의 중심은 프로젝트 운영이다.

1차에서 구현한다.

- 사용 기록 남김

왜 1차인가?

신입 부원, 공식 팀 배정, 프로젝트 참여를 운영진이 수동으로 전부 처리하면 웹의 장점이 줄어든다.

### 2.11 투표 1차

1차에서 구현한다.

- 투표 페이지
- 회장 선출 투표
- 일반 안건 투표
- 찬반 투표
- 단일 선택 투표
- 복수 선택 투표
- 복수 선택 최대 선택 개수 설정
- 익명/기명 선택
- 기본 익명
- 결과 공개 시점 선택
- 기본 종료 후 공개
- 즉시 오픈
- 예약 오픈
- 종료 시간 직접 설정
- 일반 안건 최대 30일
- 시작 전 수정 가능
- 시작 후 문항/선택지 수정 불가
- 생성자와 상위 권한자 취소 가능
- 개별 투표 기록 1년 후 삭제
- 결과 요약은 회의록/안건 기록에 보존 가능

회장 투표는 별도 규칙을 적용한다.

- 투표권자: 코봇 활동 부원 전체
- 후보: 코봇 활동 부원 누구나
- 임시 회장 지정 후 14일 후보 참여
- 15일째 투표 시작
- 투표 기간 3일
- 임시 회장 기본 후보 자동 등록
- 임시 회장 후보 사퇴 가능
- 익명 투표 고정
- 진행 중 득표수 비공개
- 종료 후 결과 공개
- 최다 득표자 당선
- 동률 시 결선 투표
- 결선도 3일
- 결선 동률 시 부회장 1명 + 공식 팀장 2명 승인 조합
- 단일 후보는 찬성/반대
- 찬성 > 반대 통과
- 반대 > 찬성 후보 모집부터 재시작
- 찬성 = 반대 부회장 1명 + 공식 팀장 2명 결정
- 당선자 3일 내 수락
- 미수락/거절 시 재투표
- 출마자가 없거나 결론이 없으면 부회장이 회장 권한 대행

왜 1차인가?

회장 유고 복구와 운영 의사결정이 웹에 들어오려면 투표의 최소 버전이 필요하다.

## 3. 2차 구현 범위

### 3.1 GitHub 연동

2차에서 구현한다.

- `Kmu-Kobot` Organization 저장소 연결
- 공개 repository README 조회
- private repository README 조회
- GitHub App 기반 서버 조회
- README 최신 commit 시각 조회
- 내부 소개서 수정 시각과 GitHub README 수정 시각 비교
- 자동/내부 고정/GitHub 고정 표시 방식
- GitHub token 프론트엔드 미노출

왜 2차인가?

프로젝트 소개에는 중요하지만, GitHub App, private repo 권한, 서버 함수까지 들어가서 1차 로그인/권한보다 복잡하다.

### 3.2 프로젝트 고급 운영

2차에서 구현한다.

- 프로젝트 팀장 양도
- 기존 팀장의 이후 상태 선택
- 공식 팀장의 프로젝트 팀장 강제 변경 요청

## 6.1 Supabase 프로젝트 리전

Supabase 프로젝트는 `Northeast Asia (Seoul)`, `ap-northeast-2` 리전으로 생성한다.

이유는 다음과 같다.

- 주요 사용자가 국민대학교 구성원으로 한국에 있다.
- 로그인, 프로필, 학번, 전화번호, 감사 기록 등 개인정보성 데이터가 많다.
- 한국 사용자 기준 응답 지연을 줄일 수 있다.
- 개인정보 처리방침에서 서버 위치와 처리 조건을 설명하기 쉽다.

이미 다른 리전으로 만든 Supabase 프로젝트가 있다면 운영 데이터가 쌓이기 전에 서울 리전 프로젝트를 새로 만들고 migration을 다시 적용한다.

현재 전달받은 Supabase URL은 다음과 같다.

```text
https://tqidhnjmjbvrzdeiqlqo.supabase.co
```

Supabase Dashboard의 `Project Settings -> General -> Region`에서 이 프로젝트가 `Northeast Asia (Seoul)`, `ap-northeast-2`인지 확인해야 한다.

## 8. 학교 전산팀 요청 메일 준비

이메일 발신 도메인 연동은 1차 개발 완료 후 학교 담당자에게 요청할 수 있다.

요청 목적은 다음과 같다.

- `kobot.kookmin.ac.kr` 공식 웹 도메인 확인
- Vercel 배포 주소와 학교 서브도메인 연결 확인
- Resend를 통한 시스템 이메일 발송 가능 여부 확인
- 가능한 경우 이메일 발신 인증 DNS 레코드 등록 요청
- 어려운 경우 학교 공식 메일을 Reply-To 또는 문의처로 사용할 수 있는지 확인

### 8.1 학교에 제공할 정보

학교 담당자에게 제공할 정보는 다음과 같다.

- 서비스명: Kobot Web
- 운영 주체: 국민대학교 Kobot 동아리
- 공식 웹 도메인: `kobot.kookmin.ac.kr`
- 배포 플랫폼: Vercel
- Vercel 프로젝트 도메인: 개발 완료 후 실제 Vercel 도메인 기입
- 요청 목적: 동아리 웹사이트 접속 및 시스템 알림 이메일 발송
- 시스템 알림 예시: 가입 승인, 프로젝트 참여 요청, 투표 알림, 연락 요청
- 이메일 발송 서비스 후보: Resend
- 발신 희망 주소 후보: `noreply@kobot.kookmin.ac.kr`
- Reply-To 희망 주소 후보: `kobot@kookmin.ac.kr`
- 개인정보 처리방침 URL: 개발 완료 후 `https://kobot.kookmin.ac.kr/privacy` 기입
- 이용약관 URL: 개발 완료 후 `https://kobot.kookmin.ac.kr/terms` 기입
- 운영 문의 이메일: `kobot@kookmin.ac.kr`
- 개인정보 문의 이메일: `kobot@kookmin.ac.kr`

### 8.2 학교에 요청할 작업

학교 담당자에게 요청할 작업은 다음과 같다.

1. `kobot.kookmin.ac.kr`이 Vercel 배포 주소로 연결되어 있는지 확인한다.
2. 연결이 필요하면 Vercel에서 안내하는 CNAME 또는 필요한 DNS 레코드를 등록한다.
3. Resend를 통해 `noreply@kobot.kookmin.ac.kr` 발신이 가능한지 검토한다.
4. 가능하면 Resend가 제공하는 SPF, DKIM, DMARC, Return-Path 관련 DNS 레코드를 등록한다.
5. 학교 정책상 외부 발송 서비스가 어렵다면, 학교 공식 SMTP 또는 Google Workspace SMTP 사용 가능 여부를 안내한다.
6. 학교 공식 메일 `kobot@kookmin.ac.kr`을 Reply-To 또는 문의처로 사용할 수 있는지 확인한다.
7. 외부 발송은 불가하지만 수신 주소는 가능하다면 `kobot@kookmin.ac.kr`을 Reply-To 또는 문의처로 사용한다.

### 8.3 개발 후 학교에 전달해야 할 DNS 값

개발 후 Resend에서 도메인 인증을 시작하면 Resend가 실제 DNS 값을 제공한다.

학교에는 Resend 화면에 표시된 값을 그대로 전달한다.

전달할 수 있는 레코드 유형은 다음과 같다.

- TXT
- CNAME
- MX

전달 항목은 다음 형식을 따른다.

| 구분 | Type | Name/Host | Value/Target | Priority | 목적 |
| --- | --- | --- | --- | --- | --- |
| Resend SPF | TXT | 개발 후 기입 | 개발 후 기입 | 해당 없음 | 발신 서버 인증 |
| Resend DKIM | CNAME 또는 TXT | 개발 후 기입 | 개발 후 기입 | 해당 없음 | 메일 서명 인증 |
| Resend Return-Path | CNAME 또는 MX | 개발 후 기입 | 개발 후 기입 | 개발 후 기입 | 반송 처리 |
| DMARC | TXT | 개발 후 기입 | 개발 후 기입 | 해당 없음 | 도메인 보호 정책 |

주의할 점은 다음과 같다.

- Resend가 제공하기 전에는 정확한 DNS 값을 미리 알 수 없다.
- 학교 담당자에게는 Resend에서 표시되는 값 그대로 복사해 전달해야 한다.
- DNS 레코드 등록 후 Resend에서 인증 완료 여부를 확인해야 한다.
- 인증이 완료되기 전까지 공식 발신 주소로 실제 운영 메일을 보내지 않는다.

### 8.4 학교 요청 메일 초안

아래 초안은 개발 완료 후 실제 Vercel 도메인과 Resend DNS 값을 채워서 보낸다.

```text

운영 주체: 국민대학교 Kobot 동아리
공식 웹 도메인: kobot.kookmin.ac.kr

1. kobot.kookmin.ac.kr 도메인이 Vercel 배포 주소로 정상 연결되어 있는지 확인 부탁드립니다.
2. 연결이 필요하다면 아래 Vercel 안내 DNS 레코드 등록을 부탁드립니다.
   - Type: [개발 완료 후 기입]
   - Name/Host: [개발 완료 후 기입]
   - Value/Target: [개발 완료 후 기입]
3. Kobot 웹사이트에서 가입 승인, 프로젝트 참여 요청, 투표 알림, 연락 요청 등의 시스템 알림 이메일을 발송하려고 합니다.
4. 이메일 발송 서비스로 Resend 사용을 검토 중이며, 가능하다면 noreply@kobot.kookmin.ac.kr 주소로 발신하고 싶습니다.
5. 이를 위해 아래 Resend 도메인 인증 DNS 레코드 등록이 가능한지 확인 부탁드립니다.
   - SPF/TXT: [개발 완료 후 Resend 값 기입]
   - DKIM: [개발 완료 후 Resend 값 기입]
   - Return-Path: [개발 완료 후 Resend 값 기입]

7. Kobot 공식 문의용 메일 주소는 kobot@kookmin.ac.kr입니다. 외부 발송 서비스 사용이 어렵다면 이 주소를 Reply-To 또는 문의처로 사용하는 방식이 가능한지 확인 부탁드립니다.

개인정보 처리방침 URL: https://kobot.kookmin.ac.kr/privacy
이용약관 URL: https://kobot.kookmin.ac.kr/terms

감사합니다.
Kobot 웹사이트 운영 담당자 드림
```
