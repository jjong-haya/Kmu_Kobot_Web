# 멤버 워크스페이스 정책

## 1. 목적

이 문서는 Kobot 웹사이트의 로그인 후 영역인 `멤버 워크스페이스`의 제품 정책, 권한 모델, 프로젝트 팀 운영, 초대 코드, GitHub 연동, 감사 기록 기준을 정리한다.

이 문서는 구현 전 기준 문서이며, 코드와 DB 설계는 이 정책을 기준으로 맞춘다.

## 2. 제품 범위

### 2.1 공개 메인 페이지

공개 메인 페이지는 발표, 포트폴리오, 시연용 화면이다.

공식 웹 도메인은 다음을 기준으로 한다.

```text
kobot.kookmin.ac.kr
```

운영 문의와 개인정보 문의 공식 메일은 다음을 기준으로 한다.

```text
kobot@kookmin.ac.kr
```

Google OAuth 동의 화면 서비스명은 다음을 기준으로 한다.

```text
KOBOT
```

2026-04-28 확인 기준 `https://kobot.kookmin.ac.kr/` 루트는 Vercel에서 정상 응답한다.

React Router 하위 경로가 직접 접속 시 404가 되지 않도록 Vercel SPA rewrite 설정을 유지한다.

개인정보 처리방침과 이용약관 URL은 다음을 기준으로 한다.

```text
https://kobot.kookmin.ac.kr/privacy
https://kobot.kookmin.ac.kr/terms
```

목표는 다음과 같다.

- Kobot의 활동, 프로젝트, 모집 정보를 보기 좋게 보여준다.
- 발표나 시연 상황에서 완성도 높은 첫인상을 제공한다.
- 로그인 없이도 핵심 소개를 볼 수 있게 한다.

### 2.2 멤버 워크스페이스

`/member/*` 아래 로그인 후 영역 전체를 `멤버 워크스페이스`라고 부른다.

목표는 다음과 같다.

- 실제 동아리원이 로그인해서 공지, 자료, 장비, 프로젝트, 스터디, 팀 활동을 관리한다.
- 회장, 부회장, 공식 팀장, 프로젝트 팀장, 일반 부원의 권한을 구분한다.
- 모든 중요한 승인, 변경, 참여, 초대 기록을 남긴다.

### 2.3 대시보드

`/member` 첫 화면은 `대시보드`라고 부른다.

대시보드는 멤버 워크스페이스 전체의 요약 화면이다.

포함할 수 있는 정보는 다음과 같다.

- 최근 공지
- 최근 자료
- 내 프로젝트
- 이번 주 일정
- 오늘 할 일
- 장비 대여 상태
- 알림

## 3. 1차 기능화 순서

멤버 워크스페이스는 최종적으로 모든 기능을 실제 운영 가능하게 만든다.

1차 실제 기능화 순서는 다음과 같다.

1. 공지
2. 멤버와 권한
3. 자료실
4. 장비 대여
5. 프로젝트
6. 이벤트와 출석
7. 스터디, Q&A, 피어리뷰
8. 관리자 도구

## 4. 공식 팀과 프로젝트 팀

### 4.1 공식 팀

공식 팀은 동아리 운영 단위다.

공식 팀 목록은 다음과 같다.

- 로봇 A팀
- 로봇 B팀
- 로봇 C팀
- 로봇 D팀
- IoT팀
- 연구팀

공식 팀은 회장, 부회장, 공식 팀장 권한과 연결된다.

### 4.2 프로젝트 팀

프로젝트 팀은 부원들이 실제 협업을 위해 생성하는 작업 단위다.

정책은 다음과 같다.

- 활동 부원은 프로젝트 팀 생성을 신청할 수 있다.
- 프로젝트 팀 생성은 승인 후 활성화된다.
- 프로젝트 팀은 공식 팀과 별개로 존재한다.
- 한 명의 부원은 공식 팀에 소속되면서 여러 프로젝트 팀에 참여할 수 있다.
- 프로젝트 팀별로 스터디 기록, 활동 기록, 자료, 회의록을 남길 수 있다.

## 5. 사용자 상태와 조직 권한

### 5.1 사용자 상태

사용자 상태는 다음과 같이 구분한다.

- 승인 대기
- 활동 부원
- 휴면
- 졸업
- 거절 또는 차단

신규 가입자는 기본적으로 `승인 대기` 상태다.

최초 로그인 사용자는 프로필 설정 단계에서 닉네임을 반드시 입력해야 한다.

닉네임을 만들기 전에는 멤버 워크스페이스에 진입할 수 없다.

최초 프로필 설정에서 실명도 필수로 입력한다.

실명은 일반 멤버에게 공개하지 않는다.

실명은 회원 승인과 운영 확인을 위해 회장, 부회장, 공식 팀장 등 승인권자 이상만 확인할 수 있다.

학번은 모든 가입자에게 필수로 받는다.

학번은 일반 멤버에게 공개하지 않고, 회원 승인과 운영 확인을 위해 승인권자 이상만 확인할 수 있다.

전화번호도 모든 가입자에게 필수로 받는다.

전화번호는 일반 멤버에게 공개하지 않는다.

1차 구현에서는 전화번호 인증을 하지 않고 입력값만 받는다.

전화번호는 회원 승인, 운영 연락, 긴급 연락, 프로젝트 내부 소통, 장비 대여 또는 행사 운영처럼 필요한 경우에 사용한다.

전화번호는 필요한 경우에만 회장, 부회장, 공식 팀장 등 승인권자 이상이 확인할 수 있다.

전화번호 열람은 같은 활동 단위에 함께 포함된 사람끼리 허용한다.

활동 단위 예시는 다음과 같다.

- 공식 팀
- 프로젝트 팀
- 스터디
- 행사 운영 그룹
- 장비 대여 또는 제작 작업 그룹

같은 활동 단위에 함께 포함된 구성원들은 내부 소통을 위해 서로의 전화번호를 확인할 수 있다.

같은 활동 단위에 포함되어 있지 않은 일반 멤버는 해당 사용자의 전화번호를 볼 수 없다.

전화번호를 보는 행위 자체는 별도 감사 기록으로 남기지 않는다.

각 활동 단위의 참여자 목록에서는 자신을 포함한 구성원 정보를 표 형태로 보여준다.

참여자 목록에 표시할 수 있는 항목은 다음과 같다.

- 역할
- 이름과 닉네임
- 연락처
- 학번
- 학과
- 소속 단과대
- 소속 공식 팀

내부 활동 단위의 참여자 목록에서는 실명을 우선 표시하고 닉네임을 보조 표시한다.

표시 예시는 다음과 같다.

```text
홍길동 (코봇러)
```

공개 페이지에서는 실명을 기본 표시하지 않고, 사용자가 선택한 이름 공개, 닉네임 공개, 익명 표시 설정을 따른다.

이 목록은 해당 활동 단위에 포함된 구성원과 운영진만 볼 수 있다.

### 5.7 프로필과 참여자 목록 표시 범위

사용자 정보 표시는 화면의 목적에 따라 구분한다.

#### 공개형 프로필 또는 기술 블로그

자신의 기술, 관심 분야, 활동 기록을 소개하는 공개형 프로필 또는 블로그형 페이지에서는 최소 정보만 표시한다.

표시할 수 있는 항목은 다음과 같다.

- 닉네임
- 기술 태그
- 자기소개
- 공개로 설정한 프로젝트 또는 글

공개형 프로필에서는 실명, 이메일, 전화번호, 학번을 기본 표시하지 않는다.

공개형 프로필에는 이메일 대신 연락 요청 기능을 제공한다.

1차 구현은 실시간 채팅이 아니라 연락 요청 보내기 형태로 시작한다.

연락 요청 정책은 다음과 같다.

- 공개형 프로필에는 `연락하기` 버튼을 둔다.
- 모든 사용자는 다른 사용자에게 연락 요청을 보낼 수 있다.
- 연락 요청에는 어떤 이유로 연락하는지 작성하게 한다.
- 연락 이유에는 최소 글자 수 제한을 두지 않는다.
- 연락 요청을 보내는 사용자는 본인이 연락 가능한 연락처를 첨부한다.
- 연락 요청에 첨부 가능한 연락처 종류는 전화번호, 이메일, 카카오톡 ID, 디스코드 ID, GitHub, 기타로 둔다.
- 연락 요청을 보내려면 연락 가능한 연락처를 최소 1개 이상 첨부해야 한다.
- 연락 요청 전송에는 기본 일일 제한을 두지 않는다.
- 연락 요청을 받은 사용자는 해당 요청을 스팸성 연락으로 신고할 수 있다.
- 스팸성 연락 신고는 회장과 부회장이 확인할 수 있다.
- 회장과 부회장은 신고 내용을 확인한 뒤 필요한 조치를 할 수 있다.
- 단순히 이유가 부족하거나 맥락 없이 연락한 경우에는 경고 조치를 기본으로 한다.
- 내용 없이 연락 요청을 반복한 경우에는 연락 기능 제한을 기본으로 한다.
- 욕설, 혐오, 성적 표현, 협박, 사칭, 악성 링크 등 이상한 내용으로 연락한 경우에는 계정 정지를 기본으로 한다.
- 스크래핑, 자동화, 봇, 매크로처럼 비정상적으로 대량 연락 요청을 보내는 경우에는 시스템이 먼저 차단한다.
- 자동 차단 대상은 회장과 부회장이 확인할 수 있다.
- 자동화 차단은 팝업 경고, 서버 측 탐지, 쿨다운, 운영진 검토 큐를 함께 사용한다.
- 10분 안에 연락 요청 3건 이상을 보내면 경고 팝업을 표시한다.
- 경고 팝업 문구는 다음을 기준으로 한다.

```text
현재 짧은 시간 내 여러 건의 연락 요청을 보내고 있습니다.
불필요한 요청으로 확인될 경우 요청 제한 또는 계정 정지 조치가 이루어질 수 있습니다.
계속 보내시겠습니까?
```

- 사용자가 경고 팝업에서 확인을 누르면 해당 요청은 전송할 수 있다.
- 10분 안에 연락 요청 6건 이상을 보내면 10분 쿨다운을 적용한다.
- 1시간 안에 연락 요청 15건 이상을 보내면 회장과 부회장 검토 큐에 자동 등록한다.
- 동일하거나 유사한 내용을 여러 사용자에게 반복 전송하면 경고 후 반복 시 쿨다운을 적용한다.
- 공백, 줄바꿈, 특수문자만 일부 바꾼 내용도 유사 내용으로 판단할 수 있다.
- 같은 프로젝트나 같은 활동 단위 안에서 필요한 연락을 보내는 경우에는 탐지 기준을 완화할 수 있다.
- 봇성 패턴이 명확한 경우에는 즉시 연락 요청 기능을 제한하고 회장과 부회장 검토 대상으로 등록한다.
- 상대방 이메일과 전화번호는 노출하지 않는다.
- 로그인한 사용자만 연락할 수 있다.
- 비로그인 사용자는 로그인 후 원래 프로필 또는 연락 흐름으로 복귀한다.
- 대시보드 사이드바에 `연락` 페이지를 둔다.
- 받은 연락 요청과 보낸 연락 요청은 `연락` 페이지에서 확인한다.
- 새 연락 요청이 도착하면 대시보드 알림에 표시한다.
- 연락 요청 상태는 `대기`, `수락`, `거절`로 구분한다.
- 연락 요청은 최대 3일 동안 대기 상태로 유지된다.
- 3일 동안 수락 또는 거절하지 않으면 자동으로 거절 처리한다.
- 자동 거절 사유는 `미응답`으로 기록하고 요청자에게 표시한다.
- 자동 거절된 요청은 다시 보낼 수 있다.
- 자동 거절 후 같은 사용자에게 다시 연락 요청을 보내려면 24시간이 지나야 한다.
- 명시적으로 거절된 요청도 다시 보낼 수 있다.
- 명시적 거절 후 같은 사용자에게 다시 연락 요청을 보내려면 7일이 지나야 한다.
- 연락 요청이 수락되면 요청자가 첨부한 연락처가 수락자에게 보이고, 수락자의 연락처도 요청자에게 공개된다.
- 연락 요청을 수락할 때 수락자는 요청자에게 공개할 연락처를 선택한다.
- 수락자는 전화번호, 이메일, 카카오톡 ID, 디스코드 ID, GitHub, 기타 중 하나 이상을 선택하거나 직접 입력할 수 있다.
- 연락 요청을 수락하려면 수락자가 공개할 연락처를 최소 1개 이상 선택해야 한다.
- 연락 요청을 수락한 뒤에는 해당 요청에 대한 연락처 공개를 취소할 수 없다.
- 이미 수락된 연락 요청이 있어도 같은 사용자에게 다시 연락 요청을 보낼 수 있다.
- 수락 이력만을 이유로 재요청 제한을 두지 않는다.
- 연락 요청이 거절되면 수락자의 연락처는 공개하지 않는다.
- 연락 요청을 거절할 때 거절 사유를 선택사항으로 입력할 수 있다.
- 입력된 거절 사유는 요청자에게 표시한다.
- 이메일 알림은 선택 기능으로만 사용하고, 이메일 주소 자체는 상대방에게 공개하지 않는다.
- 향후 필요하면 연락 요청을 실시간 채팅으로 확장할 수 있다.
- 연락 요청 내용과 첨부 연락처의 기본 보관 기간은 1년이다.
- 연락 요청을 받은 사용자는 받은 연락 요청을 삭제할 수 있다.
- 연락 요청을 보낸 사용자도 보낸 연락 요청을 삭제할 수 있다.
- 삭제는 각자 자신의 연락 페이지에서만 보이지 않게 처리한다.
- 한쪽 사용자가 삭제해도 상대방의 연락 페이지에서는 유지된다.
- 시스템 원본은 1년 보관 후 삭제한다.
- 연락 요청은 상세 내용과 첨부 연락처를 감사 기록에 복사하지 않는다.
- 감사 기록에는 연락 요청 생성, 수락, 거절, 신고, 자동 차단 같은 이벤트만 남긴다.

#### 활동 단위 참여자 목록

특정 프로젝트, 공식 팀, 스터디, 행사 운영 그룹 등 활동 단위 내부의 참여자 목록에서는 내부 소통과 운영 확인을 위해 더 많은 정보를 표시할 수 있다.

표시할 수 있는 항목은 다음과 같다.

- 역할
- 이름과 닉네임
- 연락처
- 학번
- 학과
- 소속 단과대
- 소속 공식 팀

활동 단위 참여자 목록은 해당 활동 단위 구성원과 운영진만 볼 수 있다.

### 5.2 조직 직책

조직 직책은 다음과 같이 구분한다.

- 회장
- 부회장
- 공식 팀장
- 일반 부원

### 5.3 회장 권한

회장은 최고 관리자 권한을 가진다.

회장 전용 또는 최상위 권한은 다음과 같다.

- 부회장 임명과 해임
- 공식 팀장 임명과 해임
- 주요 권한 부여와 회수
- 회장 권한 이전 요청
- 시스템 핵심 설정 변경

회장은 최상위 권한 작업을 두 방식으로 수행할 수 있다.

- 즉시 지명하고 즉시 적용
- 요청을 보내고 대상자가 수락하면 적용

두 방식 모두 기록을 남긴다.

### 5.4 부회장 권한

부회장은 회장에 준하는 전체 운영 권한을 가진다.

단, 회장 전용 최상위 임명, 회장 권한 이전, 핵심 관리자 조정 권한은 회장에게 둔다.

### 5.5 공식 팀장 권한

공식 팀장은 자기 공식 팀을 운영한다.

공식 팀장의 주요 권한은 다음과 같다.

- 자기 팀 초대 코드와 초대 링크 발급
- 공식 팀 기반 프로젝트 승인
- 개인 또는 자율 프로젝트 승인
- 프로젝트 팀장 강제 변경 요청
- 자기 팀 신규 합류 알림 수신

모든 공식 팀장은 모든 프로젝트에 대해 프로젝트 팀장 강제 변경 요청을 할 수 있다.

단, 실제 강제 변경은 회장 또는 부회장 승인 후에만 적용된다.

## 6. 권한 이전과 임명

### 6.1 회장 권한 이전

회장 권한 이전은 요청과 수락 방식으로 처리한다.

흐름은 다음과 같다.

1. 기존 회장이 새 회장 후보에게 이전 요청을 보낸다.
2. 기존 회장은 이전 완료 후 자신의 상태나 역할을 선택한다.
3. 새 회장 후보가 수락한다.
4. 회장 권한이 이전된다.
5. 기존 회장의 선택 상태가 자동 반영된다.
6. 모든 기록을 남긴다.

기존 회장이 선택할 수 있는 이전 후 상태 예시는 다음과 같다.

- 부회장
- 운영진 성격의 직책
- 일반 부원
- 휴면
- 졸업

### 6.2 공식 팀장 변경

공식 팀장 임명과 해임은 회장이 수행한다.

회장은 다음 두 방식을 선택할 수 있다.

- 즉시 지명
- 요청 후 수락

공식 팀장이 변경될 때 기존 공식 팀장의 이후 상태도 선택해야 한다.

예시는 다음과 같다.

- 해당 공식 팀 팀원으로 남기
- 다른 공식 팀으로 이동 요청
- 일반 부원
- 휴면
- 졸업

모든 변경은 기록으로 남긴다.

### 6.3 프로젝트 팀장 변경

프로젝트 팀장 변경은 두 경로로 나눈다.

#### 일반 양도

1. 현재 프로젝트 팀장이 새 팀장 후보에게 권한 양도 요청을 보낸다.
2. 현재 프로젝트 팀장은 양도 완료 후 자신의 상태를 선택한다.
3. 새 팀장 후보가 수락한다.
4. 프로젝트 팀장이 변경된다.
5. 기존 팀장의 선택 상태가 자동 반영된다.
6. 기록을 남긴다.

기존 팀장이 선택할 수 있는 상태 예시는 다음과 같다.

- 프로젝트 팀원으로 남기
- 프로젝트에서 탈퇴

#### 강제 변경 요청

현재 프로젝트 팀장이 탈퇴, 무응답, 활동 불가 상태인 경우를 위한 절차다.

1. 공식 팀장이 프로젝트 팀장 강제 변경 요청을 보낸다.
2. 회장 또는 부회장이 승인한다.
3. 프로젝트 팀장이 변경된다.
4. 기록을 남긴다.

모든 공식 팀장은 모든 프로젝트에 대해 강제 변경 요청을 보낼 수 있다.

단, 실제 변경은 회장 또는 부회장 승인 없이는 적용되지 않는다.

### 6.4 회장 유고 시 최고 관리자 복구

회장이 졸업, 탈퇴, 장기 미접속, 계정 접근 불가 등으로 최고 관리자 역할을 수행할 수 없는 경우를 대비해 복구 절차를 둔다.

회장 유고 시 최고 관리자 복구 요청은 다음 승인 조합으로 처리한다.

- 부회장 1명
- 공식 팀장 2명

부회장 1명과 공식 팀장 2명이 모두 승인하면 임시 회장 지정 절차를 진행할 수 있다.

복구 절차로 바로 새 회장을 임명하지 않는다.

임시 회장 권한 기간은 2주다.

정식 회장 임명이 필요한 경우 투표 서비스를 통해 공식 회장 임명 절차를 진행할 수 있다.

복구 요청, 승인자, 승인 시각, 사유, 지정된 임시 회장 정보는 모두 감사 기록으로 남긴다.

복구 절차는 기존 회장 권한을 우회하는 민감한 작업이므로 회장 유고 상황에서만 사용한다.

## 7. 투표 서비스

### 7.1 목적

투표 서비스는 동아리 운영상 중요한 의사결정을 웹에서 기록과 함께 진행하기 위한 기능이다.

대표 사용 사례는 다음과 같다.

- 정식 회장 임명
- 중요 운영 정책 결정
- 공식 팀장 또는 운영진 선출
- 주요 회칙 또는 운영 방식 변경
- 프로젝트 또는 활동 단위 참여자 투표
- 행사 일정, 스터디 주제, 활동 방향 같은 일반 안건 투표

### 7.2 정식 회장 임명 투표

회장 유고 복구로 임시 회장이 지정된 경우, 필요하면 정식 회장 임명을 위한 투표를 진행할 수 있다.

정식 회장 임명 투표권자는 코봇 활동 부원 전체다.

외부 프로젝트 참여자 또는 승인 대기 사용자는 정식 회장 임명 투표권을 가지지 않는다.

정식 회장 후보는 코봇 활동 부원 누구나 될 수 있다.

임시 회장이 지정된 날부터 14일 동안 후보 참여 기간을 둔다.

임시 회장 지정 후 15일째에 정식 회장 임명 투표가 열린다.

정식 회장 임명 투표 기간은 3일이다.

임시 회장은 정식 회장 후보에 기본으로 자동 등록된다.

임시 회장은 정식 회장 후보에서 사퇴할 수 있다.

임시 회장의 후보 사퇴도 기록으로 남긴다.

후보 참여 기간 동안 코봇 활동 부원은 직접 후보로 참여할 수 있다.

코봇 활동 부원은 다른 코봇 활동 부원을 후보로 추천할 수 있다.

후보 추천을 받으면 피추천자에게 알림을 보낸다.

추천된 사용자는 후보 참여를 수락해야 후보가 된다.

투표 기간, 결과 확정, 이의 제기 절차는 추가 결정이 필요하다.

투표 생성, 투표 시작, 투표 종료, 결과 확정, 권한 반영은 모두 감사 기록으로 남긴다.

투표 기록의 기본 보관 기간은 1년이다.

보관 기간이 지난 투표 기록은 삭제한다.

개별 투표 기록은 1년 후 삭제하지만, 투표 결과 요약은 안건 기록 또는 회의록에 남길 수 있다.

결과 요약에는 개별 투표자의 선택 내역을 포함하지 않는다.

투표 결과 요약에 남길 수 있는 항목은 다음과 같다.

- 안건명
- 투표 기간
- 투표권자 범위
- 총 투표수
- 결과
- 최종 상태
- 공개된 범위의 후보별 또는 선택지별 득표수

정식 회장 임명 투표는 익명 투표로 진행한다.

누가 투표했는지는 중복 투표 방지를 위해 기록할 수 있다.

누가 어떤 후보에게 투표했는지는 운영진도 확인할 수 없게 한다.

투표 결과는 투표 종료 후 공개한다.

투표 진행 중에는 중간 득표율과 후보별 득표 수를 공개하지 않는다.

정식 회장 당선자는 최다 득표자로 정한다.

정식 회장 임명 투표에는 별도 투표율 기준을 두지 않는다.

최다 득표자가 2명 이상이면 동률 후보만 대상으로 결선 투표를 진행한다.

결선 투표 기간도 3일이다.

결선 투표에서도 동률이면 부회장 1명과 공식 팀장 2명의 승인 조합으로 최종 당선자를 결정한다.

최종 결정 과정과 승인자는 감사 기록으로 남긴다.

투표 알림은 다음 채널로 보낸다.

- 대시보드 알림
- 사이드바 배지
- 이메일 알림

투표 시작, 마감 임박, 투표 종료, 결과 공개 시 알림을 보낸다.

마감 임박 알림은 마감 24시간 전과 마감 3시간 전에 보낸다.

투표 후 당선자가 확정되면 당선자에게 회장 권한 수락 요청을 보낸다.

당선자가 수락하면 회장 권한을 반영한다.

당선자가 3일 안에 수락하지 않으면 당선 포기로 처리한다.

당선자가 명시적으로 거절하면 즉시 당선 포기로 처리한다.

당선 포기 시 차순위 후보에게 바로 권한 수락 요청을 보내지 않는다.

당선자가 수락하지 않거나 거절하면 정식 회장 임명 투표를 다시 진행한다.

재투표에서도 결론이 나지 않거나 출마자가 없으면 부회장이 회장 권한을 대신 수행한다.

출마자가 나올 때까지 부회장이 회장 권한을 대신 수행할 수 있다.

정식 회장 후보가 단일 후보인 경우 찬성/반대 투표로 진행한다.

단일 후보 찬반 투표에서는 찬성이 반대보다 많으면 통과한다.

단일 후보 찬반 투표에서 반대가 찬성보다 많으면 후보 모집부터 다시 시작한다.

단일 후보 찬반 투표에서 찬성과 반대가 같으면 부회장 1명과 공식 팀장 2명의 승인 조합으로 최종 결정한다.

당선자 수락과 권한 반영은 감사 기록으로 남긴다.

### 7.3 일반 안건 투표

투표 서비스는 회장 선출 외 일반 안건에도 사용할 수 있다.

일반 안건 투표 예시는 다음과 같다.

- 회칙 변경 의견 수렴
- 행사 일정 선택
- 스터디 주제 선택
- 프로젝트 내부 의사결정
- 활동 단위 참여자 투표
- 공식 팀 내부 의사결정

일반 안건 투표는 투표 생성자가 투표권자 범위를 선택할 수 있다.

일반 안건 투표 생성 가능자는 다음과 같다.

- 회장
- 부회장
- 공식 팀장
- 프로젝트 팀장
- 스터디장

일반 안건 투표 생성자는 자기 권한 범위 안에서만 투표를 만들 수 있다.

일반 안건 투표는 생성자가 익명 투표 또는 기명 투표를 선택할 수 있다.

일반 안건 투표의 기본값은 익명 투표다.

일반 안건 투표 결과 공개 시점은 생성자가 선택할 수 있다.

선택지는 다음과 같다.

- 투표 진행 중 공개
- 투표 종료 후 공개

일반 안건 투표 결과 공개 시점의 기본값은 투표 종료 후 공개다.

투표 생성 시 오픈 방식을 선택할 수 있다.

선택지는 다음과 같다.

- 즉시 오픈
- 예약 오픈

예약 오픈을 선택하면 지정한 시간에 투표가 자동으로 시작된다.

일반 안건 투표의 종료 시간은 생성자가 직접 설정한다.

일반 안건 투표의 최대 기간은 30일이다.

1차 일반 안건 투표 유형은 다음을 지원한다.

- 찬반 투표
- 단일 선택 투표
- 복수 선택 투표

복수 선택 투표에서는 생성자가 최대 선택 개수를 설정할 수 있다.

복수 선택 투표의 기본값은 최대 선택 개수 제한 없음이다.

서술형 투표는 1차 범위에서 제외하고 이후 확장으로 둔다.

일반 안건 투표는 투표 시작 후 문항과 선택지를 수정할 수 없다.

투표가 아직 오픈되기 전에는 문항, 선택지, 투표 설정을 수정할 수 있다.

오타 또는 내용 오류가 있으면 기존 투표를 취소하고 새 투표를 생성한다.

투표 취소는 생성자와 생성자의 상위 권한자가 할 수 있다.

예시는 다음과 같다.

- 프로젝트 팀장이 만든 투표는 해당 프로젝트 팀장과 상위 운영진이 취소할 수 있다.
- 스터디장이 만든 투표는 해당 스터디장과 상위 운영진이 취소할 수 있다.
- 공식 팀장이 만든 투표는 해당 공식 팀장, 회장, 부회장이 취소할 수 있다.

투표 취소는 감사 기록으로 남긴다.

투표권자 범위 예시는 다음과 같다.

- 코봇 활동 부원 전체
- 특정 공식 팀 구성원
- 특정 프로젝트 팀 구성원
- 특정 스터디 참여자
- 특정 행사 또는 활동 단위 참여자

일반 안건 투표의 생성 권한, 익명 여부, 결과 공개 시점, 통과 기준은 안건 유형별로 추가 결정할 수 있다.

## 8. 프로젝트 생성과 승인

### 7.1 프로젝트 유형

프로젝트 생성 시 유형을 선택한다.

- 공식 팀 기반 프로젝트
- 개인 또는 자율 프로젝트

공식 팀 기반 프로젝트는 공식 팀 이름을 걸고 진행하는 프로젝트다.

개인 또는 자율 프로젝트는 새 이름으로 자유롭게 진행하는 프로젝트다.

### 7.2 승인권자

공식 팀 기반 프로젝트의 승인권자는 다음과 같다.

- 해당 공식 팀 팀장
- 회장
- 부회장

개인 또는 자율 프로젝트의 승인권자는 다음과 같다.

- 회장
- 부회장
- 모든 공식 팀장

### 7.3 승인 전 프로젝트

프로젝트는 승인 전에도 팀원을 모집할 수 있다.

정책은 다음과 같다.

- 승인 전 프로젝트는 전체 목록에 노출하지 않는다.
- 승인 전 프로젝트는 공유 링크로 접근 가능하다.
- 승인자는 프로젝트 상세에서 신청자, 참여 예정자, 소개서, 필요 역할, 기술 태그를 확인할 수 있다.
- 승인 또는 반려 기록을 남긴다.

### 7.4 프로젝트 생성 신청 필수 항목

프로젝트 생성 신청 시 필수 항목은 다음과 같다.

- 프로젝트 이름
- 프로젝트 유형
- 공개 범위
- 소개서
- 목표
- 예상 기간
- 모집 여부
- 필요 역할
- 기술 태그

### 7.5 프로젝트 공개 범위

프로젝트 공개 범위는 다음과 같이 구분한다.

- 전원 공개
- 비공개

전원 공개 프로젝트는 승인 후 활동 부원이 볼 수 있다.

비공개 프로젝트는 운영진과 해당 프로젝트 관련자만 볼 수 있다.

여기서 운영진은 다음을 뜻한다.

- 회장
- 부회장
- 공식 팀장

프로젝트 팀장은 운영진이 아니다.

프로젝트 팀장은 자기 프로젝트만 관리하고 볼 수 있다.

## 9. 프로젝트 열람 권한

### 8.1 비공개 프로젝트 전체 열람 가능자

비공개 프로젝트 전체를 볼 수 있는 사람은 다음과 같다.

- 회장
- 부회장
- 공식 팀장
- 해당 프로젝트 팀장
- 해당 프로젝트 팀원

### 8.2 참여 신청자 권한

참여 신청자는 프로젝트 전체 자료를 볼 수 없다.

참여 신청자는 다음만 볼 수 있다.

- 프로젝트 README
- 프로젝트 소개서
- 참여 신청 상태

팀 내부 자료는 승인된 프로젝트 팀원 이상만 볼 수 있다.

## 10. 프로젝트 소개서와 GitHub README

### 9.1 소개서 소스

프로젝트 소개서는 두 소스를 지원한다.

- 코봇 웹 내부 작성 소개서
- GitHub README

GitHub 저장소 연결은 필수가 아니다.

연구팀 프로젝트처럼 GitHub 저장소가 없는 활동도 생성할 수 있다.

### 9.2 소개서 표시 방식

프로젝트 팀장은 소개서 표시 방식을 선택할 수 있다.

- 자동
- 내부 소개서 고정
- GitHub README 고정

`자동`은 내부 소개서 수정 시각과 GitHub README 수정 시각을 비교해 더 최근에 수정된 쪽을 표시한다.

GitHub README 수정 시각은 README 파일의 최신 commit 시각을 기준으로 한다.

### 9.3 소개서 표시 방식 변경

프로젝트 팀원은 소개서 표시 방식 변경을 요청할 수 있다.

프로젝트 팀장이 승인하거나 반려한다.

프로젝트 팀장이 직접 변경하는 것도 가능하다.

요청, 승인, 반려, 변경 기록은 모두 남긴다.

## 11. GitHub 연동 정책

### 10.1 1차 범위

1차 GitHub 연동은 동아리 GitHub Organization 안의 저장소만 공식 지원한다.

Organization 이름은 다음과 같다.

```text
Kmu-Kobot
```

1차 공식 지원 저장소 URL 형태는 다음과 같다.

```text
https://github.com/Kmu-Kobot/{repo}
```

개인 GitHub 저장소 연결은 1차 범위에서 제외하고, 이후 확장으로 둔다.

### 10.2 공개 저장소와 비공개 저장소

공개 저장소와 비공개 저장소를 모두 지원한다.

비공개 저장소 README를 코봇 웹에서 보여주려면 GitHub App과 서버 함수가 필요하다.

권장 구조는 다음과 같다.

```text
브라우저
→ 코봇 웹 로그인 상태만 보유
→ GitHub token 없음

서버 또는 Supabase Edge Function
→ 현재 사용자 권한 확인
→ GitHub App 권한으로 README 조회
→ README 결과만 브라우저에 반환
```

### 10.3 프로젝트 팀장이 해야 할 일

프로젝트 팀장이 해야 할 일은 다음 정도로 단순화한다.

1. `Kmu-Kobot` Organization 안에 GitHub 저장소를 만든다.
2. `README.md`를 작성한다.
3. 코봇 웹 프로젝트 설정에 GitHub 저장소 URL을 입력한다.
4. 연결 확인을 누른다.
5. README가 코봇 웹에서 표시되는지 확인한다.

GitHub App 설치와 권한 관리는 운영진 또는 개발자가 Organization 단위로 한 번 처리한다.

프로젝트 팀장이 GitHub App 구조를 이해해야 하는 UX는 피한다.

## 12. 팀원 모집 공유 페이지

### 11.1 목적

팀원 모집 공유 페이지는 카카오톡 공지방이나 동아리 내부 채널에 공유하기 위한 페이지다.

목표는 다음과 같다.

- 프로젝트를 보기 좋게 소개한다.
- 필요한 역할과 기술 태그를 강조한다.
- 부원이 자신이 참여할 수 있는지 빠르게 판단하게 한다.
- 승인자 검토 화면과 분리한다.

### 11.2 포함 정보

팀원 모집 공유 페이지에는 다음 정보를 보여준다.

- 프로젝트 이름
- 한 줄 소개
- 목표
- 프로젝트 유형
- 공개 범위
- 예상 기간
- 모집 여부
- 필요한 역할
- 기술 태그

- 공유용 카드 또는 미리보기

프로젝트 생성 시 모집 대상 범위를 선택한다.

- 동아리 내부 모집
- 동아리 외부 사용자도 모집

승인 전 프로젝트도 모집 공유 페이지를 가질 수 있다.

전체 프로젝트 목록에는 승인된 프로젝트만 노출한다.

승인 전 프로젝트는 공유 링크로 접근한다.

## 13. 외부 사용자와 회원가입

### 12.1 외부 사용자 모집

프로젝트 생성 시 동아리 외부 사용자도 모집할 수 있다.

여기서 외부 사용자는 기본적으로 `코봇 동아리 외부 사용자`를 뜻한다.

즉, 학교 외부 인원이 아니라 코봇 정식 부원이 아닌 학교 구성원을 포함한다.

셀프 회원가입은 학교 구성원을 기본 대상으로 한다.

학교 구성원은 Google 첫 로그인 후 학교 메일 여부를 확인한다.

학교 메일 기준은 다음과 같다.

```text
kookmin.ac.kr
```

학교 메일 사용자는 회원가입 후 프로젝트 참여 신청을 할 수 있다.

학교 구성원이 아니지만 프로젝트 참여가 필요한 예외 인원은 운영자가 수동으로 계정을 생성한다.

학교 밖 예외 계정 생성 가능자는 다음과 같다.

- 회장
- 부회장

수동 생성 계정은 사용자가 이후 프로필에서 아이디, 비밀번호, 기본 정보를 수정할 수 있게 한다.

### 12.2 프로필 필수 정보

회원가입 또는 프로필 설정에서 다음 정보를 입력할 수 있게 한다.

- 단과대
- 학과
- 소속 동아리

이 정보는 코봇 인원과 외부 사용자 모두에게 동일하게 적용한다.

학교 밖 예외 인원도 운영자가 수동 생성한 계정으로 로그인한 뒤 동일한 프로필 정보를 수정할 수 있다.

### 12.3 신규 가입 기본 상태

신규 가입자는 기본적으로 승인 대기 상태다.

활동 부원이 되려면 다음 중 하나가 필요하다.

- 회장, 부회장, 공식 팀장의 승인
- 코봇 멤버 인증 코드 입력
- 공식 팀 초대 코드 또는 초대 링크 사용

## 14. 참여 코드와 초대 링크

### 13.1 코드 종류

참여 코드와 초대 링크는 다음 종류로 나눈다.

- 코봇 멤버 인증 코드
- 공식 팀 초대 코드
- 공식 팀 초대 링크
- 프로젝트 참여 코드
- 프로젝트 참여 링크

### 13.2 기본 유효기한

모든 코드와 링크는 유효기한을 가진다.

기본 유효기한은 다음과 같다.

```text
5일
```

생성 시 유효기한을 직접 정할 수 있다.

### 13.3 코봇 멤버 인증 코드

코봇 멤버 인증 코드는 승인 대기 사용자를 코봇 활동 부원으로 전환하는 코드다.

발급 가능자는 다음과 같다.

- 회장
- 부회장

공식 팀장은 코봇 전체 멤버 인증 코드를 발급할 수 없다.

### 13.4 공식 팀 초대 코드와 링크

공식 팀 초대 코드와 링크는 사용자를 코봇 활동 부원으로 전환하고, 동시에 해당 공식 팀 소속으로 등록한다.

발급 가능자는 다음과 같다.

- 회장: 모든 공식 팀 초대 코드와 링크 발급 가능
- 부회장: 모든 공식 팀 초대 코드와 링크 발급 가능
- 공식 팀장: 자기 팀 초대 코드와 링크만 발급 가능

공식 팀 초대 코드 또는 링크를 사용하면 즉시 승인된다.

팀장에게는 누가 들어왔는지 알림을 보낸다.

사용 기록을 남긴다.

### 13.5 프로젝트 참여 코드와 링크

프로젝트 팀장은 프로젝트 참여 코드와 링크를 발급할 수 있다.

사용자가 프로필 설정창에서 프로젝트 참여 코드를 입력하거나 링크를 통해 들어오면 해당 프로젝트에 참여된다.

프로젝트 참여 코드와 링크는 해당 프로젝트 참여 권한만 부여한다.

프로젝트 참여 코드와 링크만으로 코봇 전체 활동 부원 권한이 부여되지는 않는다.

코봇 전체 활동 부원 권한은 코봇 멤버 인증 코드, 공식 팀 초대 코드, 공식 팀 초대 링크, 또는 운영진 승인을 통해 별도로 부여한다.

프로젝트 참여 코드가 사용된 경우에도 기록을 남긴다.

## 15. 프로젝트 팀원 관리

### 14.1 참여 신청

프로젝트 참여 신청은 기록으로 남긴다.

기록에는 다음을 포함한다.

- 누가 신청했는지
- 언제 신청했는지
- 어떤 경로로 신청했는지
- 누가 승인 또는 반려했는지
- 승인 또는 반려 시각

### 14.2 프로젝트 팀원 탈퇴

프로젝트 팀원은 자유 탈퇴가 아니라 탈퇴 요청을 보낸다.

프로젝트 팀장이 탈퇴 요청을 승인하거나 반려한다.

탈퇴 요청, 승인, 반려 기록을 남긴다.

### 14.3 프로젝트 팀장 탈퇴

프로젝트 팀장은 팀장 권한을 다른 사람에게 넘기기 전까지 프로젝트에서 탈퇴할 수 없다.

프로젝트 팀장 없는 프로젝트 상태는 허용하지 않는다.

### 14.4 탈퇴 후 산출물 보존

프로젝트 소개서, 스터디 기록, 회의록, 팀 활동 기록, 자료 설명, 댓글 등 팀 산출물은 탈퇴로 인해 자동 삭제하지 않는다.

팀 산출물은 프로젝트 이력과 동아리 활동 기록을 유지하기 위해 보존한다.

탈퇴 후 작성자 표시는 실명, 이메일, 학번, Google 프로필, 아이디를 제거하고 다음 최소 정보만 남긴다.

- 닉네임
- 당시 공식 팀 소속
- 당시 프로젝트 역할
- 당시 조직 역할

표시 예시는 다음과 같다.

```text
코봇러 · 로봇 A팀 · 프로젝트 팀장 · 일반 부원
```

닉네임과 역할 정보도 다른 정보와 결합해 개인을 알아볼 수 있으면 개인정보에 해당할 수 있으므로, 회원가입과 콘텐츠 작성 화면에서 탈퇴 후 산출물 보존 및 작성자 표시 기준을 미리 안내한다.

회원가입 시 일괄 고지만으로 모든 공개 범위가 자동 허용되는 것으로 처리하지 않는다.

멤버 워크스페이스 내부 표시와 외부 공개 페이지 표시는 구분한다.

공개 페이지는 프로젝트 중심으로 보여주며, 기본적으로 누가 했는지보다 어떤 프로젝트를 했는지를 우선한다.

멤버 워크스페이스 내부에서는 항상 닉네임으로 표시한다.

내부 페이지에서는 익명 표시를 허용하지 않는다.

닉네임은 현재 활동 사용자 기준으로 중복을 허용하지 않는다.

닉네임 중복 검사는 대소문자를 구분하지 않는다.

탈퇴한 사용자의 닉네임은 재사용할 수 있다.

이전 닉네임도 재사용할 수 있다.

사용자 식별은 닉네임이 아니라 내부 사용자 ID를 기준으로 한다.

회장과 부회장은 감사 기록에서 내부 사용자 ID 기준으로 실제 행위자를 확인할 수 있다.

닉네임 변경은 7일에 1번만 허용한다.

닉네임 변경은 금칙어와 부적절한 닉네임 검사를 통과해야 적용된다.

닉네임 길이는 2자 이상 12자 이하로 제한한다.

닉네임에 허용하는 문자는 한글, 영문, 숫자, 공백이다.

닉네임 입력 시 공백은 허용한다.

닉네임 저장 시 공백은 `_`로 변환해 저장한다.

닉네임 화면 표시 시 저장된 `_`는 공백처럼 표시한다.

사용자가 직접 `_`를 입력하는 것은 허용하지 않는다.

닉네임 앞뒤 공백은 제거하고, 연속 공백은 한 칸으로 정리한다.

검사 대상 예시는 다음과 같다.

- 욕설
- 혐오 표현
- 성적 표현
- 타인 실명 사칭
- 회장, 부회장, 공식 팀장 등 운영진 사칭
- Kobot 공식 계정으로 오해될 수 있는 이름
- 지나치게 긴 닉네임
- 읽기 어렵거나 식별이 어려운 반복 문자

닉네임이 변경되어도 기존 글, 프로젝트 기록, 스터디 기록, 회의록, 참여 기록 등 과거 산출물의 작성자 표시는 작성 당시 닉네임으로 유지한다.

이를 위해 산출물과 기록은 작성자 사용자 ID와 작성 당시 닉네임 스냅샷을 함께 저장한다.

작성자 프로필을 열면 현재 닉네임과 닉네임 변경 이력을 볼 수 있게 한다.

일반 멤버 프로필에는 닉네임 변경 이력을 최근 5개까지만 표시한다.

전체 닉네임 변경 이력은 회장과 부회장이 감사 기록에서 확인할 수 있다.

닉네임 변경 시 사용자에게 이전 닉네임을 일반 멤버에게 비공개 처리할지 묻는다.

이전 닉네임 비공개를 선택하면 일반 멤버 프로필에는 해당 이전 닉네임을 숨긴다.

이전 닉네임 공개 여부는 프로필 설정에서 언제든 다시 변경할 수 있다.

단, 회장과 부회장은 감사와 운영 목적으로 비공개 처리된 이전 닉네임도 확인할 수 있다.

닉네임 변경 이력 표시 예시는 다음과 같다.

```text
alpha - Beta - gamma(현)
```

일반 멤버에게 비공개 처리된 이전 닉네임은 다음처럼 표시할 수 있다.

```text
비공개 - Beta - gamma(현)
```

단, 감사 기록의 무결성을 위해 닉네임 변경 이벤트에는 이전 닉네임과 새 닉네임을 모두 남긴다.

공개 페이지에서 개인 이름을 어떻게 표시할지는 각 사용자가 프로필 설정에서 선택한다.

선택지는 다음과 같다.

- 이름 공개
- 닉네임 공개
- 익명 표시

공개 페이지의 기본값은 `익명 표시`다.

탈퇴 요청 시에도 공개 페이지의 기존 산출물 작성자 표시 방식을 다시 선택하게 한다.

탈퇴자가 별도로 선택하지 않으면 공개 페이지에서는 익명 표시를 기본값으로 한다.

탈퇴 후에도 멤버 워크스페이스 내부에서는 프로젝트 이력 확인을 위해 작성 당시 닉네임, 당시 팀, 당시 역할을 최소 표시한다.

개인이 안전, 사생활 침해, 명예훼손, 잘못된 개인정보 노출 등의 사유로 특정 산출물의 작성자 표시 제거를 요청할 수 있는 절차를 둔다.

## 16. 프로젝트 팀장 임시 권한 위임

### 15.1 기본 정책

프로젝트 팀에는 부팀장 역할을 두지 않는다.

대신 프로젝트 팀장이 일정 기간 동안 팀장 권한 일부를 임시 위임할 수 있다.

임시 위임은 기한 제한이 필수다.

최대 기간은 다음과 같다.

```text
7일
```

2주 이상 필요하면 7일 단위로 다시 갱신해야 한다.

### 15.2 활성화 조건

임시 권한 위임은 다음 흐름으로 활성화된다.

1. 프로젝트 팀장이 팀원에게 임시 위임 요청을 보낸다.
2. 위임받는 팀원이 수락한다.
3. 임시 권한이 활성화된다.
4. 종료일이 지나면 자동 만료된다.

요청, 수락, 거절, 만료 기록을 모두 남긴다.

### 15.3 임시 위임 가능 권한

임시 위임자는 운영 업무를 처리할 수 있다.

허용 권한은 다음과 같다.

- 팀원 초대
- 참여 신청 승인과 반려
- 팀원 탈퇴 요청 승인과 반려
- 팀 내부 자료 추가와 수정
- 스터디 기록 추가와 수정
- 회의록 추가와 수정
- GitHub README 동기화 요청
- 프로젝트 상태 변경 요청
- 공지 또는 안내 작성
- 팀원 활동 기록 확인

### 15.4 임시 위임 불가 권한

임시 위임자는 핵심 설정, 소유권, 삭제 권한을 가질 수 없다.

불가 권한은 다음과 같다.

- 프로젝트 팀장 변경
- 임시 권한 재위임
- 프로젝트 삭제
- 프로젝트 폐기 또는 아카이브
- GitHub 저장소 변경
- 공개 범위 변경
- 프로젝트 이름 변경
- 프로젝트 유형 변경
- 팀원 강제 제외
- 내부 자료 전체 삭제
- 승인 신청 주요 내용 변경

## 19. 알림 정책

### 19.1 1차 알림 채널

1차 알림 채널은 다음과 같다.

- 웹 알림
- 사이드바 배지
- 이메일 알림

이메일 발송 서비스는 1차 구현에서 Resend를 사용한다.

공식 웹 도메인은 `kobot.kookmin.ac.kr`로 둔다.

다만 Resend 발신 주소로 `@kookmin.ac.kr` 또는 `@kobot.kookmin.ac.kr`을 사용하려면 학교가 해당 도메인 또는 서브도메인의 이메일 발신 인증 DNS 레코드 설정을 허용해야 한다.

학교 DNS 협조가 어렵다면 1차에서는 별도 발신 도메인을 준비하거나, 학교 공식 메일은 Reply-To 또는 문의처로만 사용하는 방식을 검토한다.

운영 문의와 개인정보 문의 공식 메일은 `kobot@kookmin.ac.kr`을 사용한다.

이메일 발신 도메인 연동은 1차 개발 완료 후 학교 담당자에게 DNS 레코드 등록 요청 메일을 보내 진행한다.

카카오톡 알림과 문자 알림은 1차 범위에서 제외한다.

이메일 알림은 중요 알림과 일반 알림으로 나눈다.

중요 알림은 기본 켜짐으로 둔다.

사용자는 중요 이메일 알림도 끌 수 있다.

중요 이메일 알림을 끌 때는 중요한 승인, 투표, 권한 요청을 놓칠 수 있다는 경고 문구를 표시한다.

일반 알림은 사용자가 선택해서 켤 수 있게 한다.

중요 알림 예시는 다음과 같다.

- 멤버 승인 요청
- 멤버 승인 또는 반려
- 프로젝트 참여 승인 또는 반려
- 프로젝트 팀장 양도 요청
- 임시 권한 위임 요청
- 연락 요청
- 투표 시작
- 투표 마감 임박
- 회장 권한 수락 요청
- 스팸 신고 조치

### 19.4 웹 알림 상태

웹 알림은 읽음과 안읽음 상태를 가진다.

사용자는 개별 알림을 읽음 처리할 수 있다.

사용자는 전체 알림을 읽음 처리할 수 있다.

사용자는 개별 웹 알림을 삭제할 수 있다.

사용자는 전체 웹 알림을 삭제할 수 있다.

웹 알림 삭제는 사용자 화면에서만 보이지 않게 처리한다.

웹 알림 삭제가 감사 기록이나 관련 원본 기록을 삭제하지는 않는다.

웹 알림의 기본 보관 기간은 1년이다.

보관 기간이 지난 웹 알림은 삭제한다.

## 20. 로그인 페이지 방향

로그인 페이지는 첨부 레퍼런스처럼 좌측 비주얼 패널과 우측 로그인 폼 구조를 사용한다.

모바일에서는 단일 컬럼으로 자연스럽게 접힌다.

좌측 영역은 단순 장식 이미지가 아니라 로그인과 가입 정책을 설명하는 안내 영역으로 사용한다.

좌측 안내 영역에는 다음 내용을 명확히 설명한다.

- 첫 로그인은 Google 학교 계정으로 진행한다.
- 기본적으로 `kookmin.ac.kr` 학교 메일만 셀프 가입할 수 있다.
- 코봇 정식 부원이 아니어도 학교 구성원은 프로젝트 참여 신청이 가능하다.
- 학교 밖 예외 인원은 회장 또는 부회장이 수동 생성한 계정으로 접근한다.
- 초대 코드나 초대 링크가 있으면 로그인 후 자동으로 이어진다.
- 로그인 후 아이디를 생성하면 이후 아이디/비밀번호로도 로그인할 수 있다.

로그인 페이지는 다음 요소를 포함할 수 있다.

- Kobot 로고
- 서비스 소개 문구
- Google 로그인
- 아이디와 비밀번호 로그인
- 참여 코드 입력 안내
- 회원가입 또는 모집 페이지 이동

로그인 정책은 다음을 기준으로 한다.

- 신규 가입자는 기본적으로 승인 대기 상태다.
- 첫 로그인은 Google 로그인으로만 시작한다.
- 첫 로그인 후 이메일이 `kookmin.ac.kr`로 끝나는지 확인한다.
- `kookmin.ac.kr`로 끝나지 않는 Google 계정은 멤버 워크스페이스 기능에 접근할 수 없다.
- 학교 메일이 아닌 계정으로 로그인한 사용자는 학교 메일만 가능하다는 안내 페이지만 볼 수 있다.
- 학교 메일이 아닌 계정 안내 페이지에는 회원 탈퇴 및 재가입 버튼을 제공한다.
- 회원 탈퇴 및 재가입 버튼을 누르면 현재 잘못 가입한 계정의 탈퇴 요청을 생성한다.
- 학교 메일이 아닌 계정의 탈퇴 요청은 회장 또는 부회장이 승인한다.
- 탈퇴 요청, 승인, 반려, 처리 결과는 기록으로 남긴다.
- 탈퇴가 승인되면 다시 Google 로그인 흐름을 시작할 수 있게 한다.
- 학교 이메일이 확인된 사용자에게 아이디 생성 안내 팝업을 띄울 수 있다.
- 사용자가 아이디 생성을 선택하면 이후 아이디/비밀번호 로그인도 사용할 수 있다.
- 아이디 로그인 선택 후에도 Google 메일 로그인은 계속 사용할 수 있다.
- 사용자가 아이디 생성 안내 팝업에서 `다시 보지 않기`를 선택하면 이후 팝업을 띄우지 않는다.
- 참여 코드나 초대 링크를 통해 승인 흐름을 단축할 수 있다.

### 18.1 진입 의도 보존

카카오톡 인앱 브라우저, 일반 모바일 브라우저, 데스크톱 브라우저, 외부 공유 링크에서 멤버 워크스페이스 링크를 열었을 때, 사용자가 처음 하려던 작업을 로그인, 회원가입, 프로필 설정, 승인 대기 흐름 이후에도 잃지 않아야 한다.

이 정책을 `진입 의도 보존`이라고 부른다.

기본 흐름은 다음과 같다.

```text
사용자가 공유 링크 클릭
→ 로그인 필요 여부 확인
→ 로그인 필요하면 진입 의도 생성
→ /login?intent={진입_의도_ID} 로 이동
→ Google 로그인
→ /auth/callback 에서 세션 확인
→ 진입 의도 복원
→ 필요하면 프로필 설정 또는 승인 흐름 처리
→ 처음 하려던 작업으로 복귀
```

예시는 다음과 같다.

```text
/projects/join/abc123
→ 진입 의도 저장
→ /login?intent=xyz789
→ Google 로그인
→ /auth/callback
→ /projects/join/abc123
```

진입 의도에는 단순 URL뿐 아니라 사용자가 하려던 작업 정보를 함께 저장할 수 있다.

예시는 다음과 같다.

```json
{
  "type": "project_join",
  "returnPath": "/projects/join/abc123",
  "projectId": "computing-project",
  "inviteCode": "abc123",
  "createdAt": "2026-04-28T14:00:00",
  "expiresAt": "2026-04-28T15:00:00"
}
```

브라우저에는 민감한 정보를 직접 저장하지 않고, 가능한 한 진입 의도 ID만 저장한다.

중요한 코드, 프로젝트 ID, 초대 상태, 권한 판단은 서버 또는 DB 기준으로 검증한다.

진입 의도의 복귀 경로는 반드시 내부 경로만 허용한다.

허용 예시는 다음과 같다.

- `/member`
- `/member/projects/123`
- `/projects/join/abc123`

차단 예시는 다음과 같다.

- `https://external-site.com`
- `//external-site.com`
- 빈 값

공유 링크, 초대 링크, 참여 코드 링크는 로그인 후에도 원래 목적지가 유지되어야 한다.

프로필 설정, 아이디 생성 안내, 승인 대기, 참여 코드 입력 같은 중간 흐름이 끼어도 진입 의도는 유지되어야 한다.
