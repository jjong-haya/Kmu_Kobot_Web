## 1. 현재 확정 기준

### 1.1 감사 기록 보존 기간

멤버 워크스페이스 감사 기록의 기본 보존 기간은 다음과 같다.

```text
1년
```

1년 보관 대상 예시는 다음과 같다.

- 회원가입
- 로그인과 로그인 실패
- 학교 메일이 아닌 계정 접속
- 탈퇴 요청, 승인, 반려
- 멤버 승인, 반려
- 권한 변경
- 공식 팀 소속 변경
- 프로젝트 생성 신청, 승인, 반려
- 프로젝트 팀원 참여 신청, 승인, 반려
- 초대 코드와 초대 링크 발급, 사용, 만료, 폐기
- GitHub 저장소 연결과 변경
- README 동기화
- 진입 의도 생성, 복원, 만료

보존 기간이 지난 기록은 기본적으로 삭제한다.

비식별 운영 통계로 별도 보관하지 않는다.

### 1.2 셀프 가입 대상

셀프 회원가입은 기본적으로 국민대학교 학교 메일 사용자를 대상으로 한다.

```text
kookmin.ac.kr
```

학교 메일이 아닌 Google 계정으로 로그인한 사용자는 멤버 워크스페이스 기능에 접근하지 못하고, 학교 메일만 가능하다는 안내 페이지를 본다.

학교 외부 예외 인원은 회장 또는 부회장이 수동으로 계정을 생성한다.

## 2. 핵심 법적 근거 요약

### 2.1 개인정보 처리방침

개인정보를 처리하는 웹서비스는 개인정보 처리방침을 수립하고 공개하는 것을 기본 원칙으로 한다.

개인정보 처리방침에는 다음 항목이 들어가야 한다.

- 개인정보 처리 목적
- 개인정보 처리 항목
- 개인정보 보유 및 이용 기간
- 개인정보 파기 절차와 방법
- 개인정보 제3자 제공에 관한 사항
- 개인정보 처리위탁에 관한 사항
- 개인정보 국외 이전에 관한 사항
- 정보주체와 법정대리인의 권리와 행사 방법
- 개인정보 보호책임자 또는 개인정보 문의처
- 개인정보 안전성 확보조치
- 쿠키 등 자동 수집 장치의 설치, 운영, 거부에 관한 사항
- 처리방침 시행일과 변경 이력

Kobot은 로그인, 프로필, 권한 승인, 프로젝트, GitHub 연동을 다루므로 개인정보 처리방침을 공개 페이지 푸터와 로그인/회원가입 화면에서 항상 접근 가능하게 둔다.

### 2.2 개인정보 수집 및 이용 동의

회원가입, 프로필 설정, 프로젝트 참여 신청, GitHub 연동처럼 개인정보를 수집하는 지점에서는 사용자가 이해할 수 있게 안내해야 한다.

동의를 받을 때 안내해야 할 핵심 항목은 다음과 같다.

- 수집 및 이용 목적
- 수집하는 개인정보 항목
- 보유 및 이용 기간
- 동의 거부권
- 동의 거부 시 불이익

필수 항목과 선택 항목은 분리한다.

선택 항목 동의를 거부했다는 이유만으로 핵심 서비스 이용을 막지 않는다. 단, 선택 항목이 필요한 부가 기능은 제한될 수 있다.

### 2.3 보유 기간과 파기

개인정보는 보유 기간이 지나거나 처리 목적이 달성되어 더 이상 필요하지 않게 되면 지체 없이 파기하는 것을 기준으로 한다.

Kobot에서는 다음 원칙을 적용한다.

- 계정 기본 정보는 회원 탈퇴 또는 회원 자격 종료 시까지 보관한다.
- 감사 기록은 운영 안정성과 분쟁 대응을 위해 1년 보관한다.
- 1년이 지난 감사 기록은 삭제한다.
- 법령이나 학교 규정상 별도 보존이 필요한 경우 해당 근거와 기간을 개인정보 처리방침에 적는다.
- GitHub private repository에서 가져온 README나 메타데이터는 필요한 범위만 사용하고, 불필요하게 장기 저장하지 않는다.

### 2.4 국외 이전, 위탁, 제3자 제공

Google OAuth, Supabase, GitHub 같은 외부 서비스를 쓰면 개인정보 처리위탁, 제3자 제공, 국외 이전 여부를 검토해야 한다.

Kobot 문서에서는 외부 서비스별로 다음 표를 관리한다.

| 서비스 | 사용 목적 | 처리되는 정보 | 위탁/제공/국외 이전 여부 | 보유 기간 | 비고 |
| --- | --- | --- | --- | --- | --- |
| Google OAuth | 로그인, 학교 메일 확인 | 이메일, 이름, 프로필 이미지, Google 계정 식별자 | 확인 필요 | 계정 유지 기간 | 최초 로그인 기준 |
| Supabase | 인증, DB, 세션 | 계정, 프로필, 권한, 감사 기록 | 서울 리전 사용 시 국내 저장 기준, 서비스 제공 과정의 처리위탁/국외 접근 가능성은 약관과 DPA 확인 필요 | 항목별 정책 | `Northeast Asia (Seoul)`, `ap-northeast-2` 사용 |
| GitHub | 프로젝트 README 연동 | 저장소 URL, README, commit 시각, private repo 접근 결과 | 확인 필요 | 연결 해제 또는 프로젝트 삭제 시까지 | token은 서버에서만 관리 |
| Resend | 이메일 알림 발송 | 이메일 주소, 알림 제목, 알림 발송 메타데이터 | 확인 필요 | 알림 발송 목적 달성 시까지 | 1차 이메일 발송 서비스 |

실제 배포 전에는 각 서비스의 계약, 리전, 데이터 처리 조건을 확인해 표를 확정한다.

Supabase 프로젝트는 한국 사용자와 개인정보 처리 부담을 고려해 `Northeast Asia (Seoul)`, `ap-northeast-2` 리전으로 생성한다.

이미 다른 리전으로 만든 Supabase 프로젝트가 있다면, 운영 데이터가 쌓이기 전 서울 리전 프로젝트를 새로 만들고 migration을 다시 적용하는 방식을 우선한다.

현재 전달받은 Supabase URL은 `https://tqidhnjmjbvrzdeiqlqo.supabase.co`다.

해당 프로젝트가 서울 리전인지 Supabase Dashboard의 `Project Settings -> General -> Region`에서 확인해야 한다.

### 2.5 친목 단체 예외에 대한 판단

개인정보 보호법에는 동창회, 동호회 등 친목 단체 운영 목적의 개인정보 처리에 대해 일부 조항 적용 예외가 존재할 수 있다.

하지만 Kobot 웹은 단순 연락망이 아니라 다음 기능을 가진다.

- 공개 홍보 페이지
- Google OAuth 로그인
- 회원 승인과 권한 관리
- 프로젝트 모집과 참여 신청
- 외부 사용자 또는 예외 계정
- GitHub 비공개 저장소 README 연동
- 자료, 장비, 공지, 감사 기록

따라서 예외만 믿기보다 일반 웹서비스 수준의 개인정보 처리방침과 동의 절차를 두는 것을 기준으로 한다.

### 3.2 개인정보 수집 및 이용 안내

개인정보 처리방침과 별개로, 실제 수집이 일어나는 화면에서 짧게 보여주는 안내다.

필요 위치는 다음과 같다.

- Google 첫 로그인 화면
- 아이디 생성 안내 팝업
- 프로필 설정 화면
- 프로젝트 참여 신청 화면
- 프로젝트 생성 화면
- GitHub 저장소 연결 화면
- 학교 메일이 아닌 계정의 탈퇴 및 재가입 요청 화면

### 3.3 서비스 이용약관

비영리 동아리 사이트라고 해서 모든 경우에 이용약관이 무조건 법정 필수라고 단정하기는 어렵다.

하지만 Kobot은 회원제 서비스이고 권한, 자료, 장비, 프로젝트, 외부 연동이 있으므로 이용약관 또는 회원 운영정책을 공개하는 것을 기준으로 한다.

포함 항목은 다음과 같다.

- 서비스 목적
- 회원 자격과 가입 승인
- 계정 관리 책임
- 금지 행위
- 게시물과 프로젝트 콘텐츠 권리
- 자료실과 장비 대여 책임
- 프로젝트 참여와 탈퇴
- GitHub 연동 책임
- 서비스 제한, 정지, 탈퇴
- 책임 제한
- 문의처

### 3.4 회원 운영정책

서비스 이용약관보다 동아리 내부 운영에 가까운 문서다.

포함 항목은 다음과 같다.

- 회장, 부회장, 공식 팀장, 프로젝트 팀장, 일반 부원 권한
- 가입 승인과 반려
- 공식 팀 배정
- 프로젝트 팀 생성과 승인
- 프로젝트 팀장 변경
- 임시 권한 위임
- 감사 기록 열람 범위
- 탈퇴 요청과 승인
- 잘못 가입한 계정의 재가입 절차

### 3.5 쿠키 및 자동 수집 장치 안내

현재 구조상 로그인 세션, Supabase 인증 저장소, 사이드바 상태 같은 자동 저장 정보가 생길 수 있다.

광고 추적이나 맞춤형 행태정보 분석이 없다면 별도 쿠키 배너까지는 과할 수 있지만, 개인정보 처리방침에는 최소한 다음을 적는다.

- 어떤 자동 수집 장치를 쓰는지
- 무엇을 위해 쓰는지
- 사용자가 어떻게 거부하거나 삭제할 수 있는지

Google Analytics, Meta Pixel, 광고 추적, 행태정보 기반 추천 기능을 추가하면 별도 고지와 거부 수단을 다시 검토한다.

### 3.6 GitHub 연동 및 저장소 공개 정책

Kobot은 GitHub README를 프로젝트 소개서로 가져올 수 있으므로 별도 정책이 필요하다.

포함 항목은 다음과 같다.

- 1차 공식 지원 범위는 `Kmu-Kobot` Organization 저장소
- public repository README 표시 기준
- private repository README 표시 기준
- private repository 정보는 권한 있는 사용자에게만 표시
- GitHub token은 프론트엔드에 저장하지 않음
- private README는 검색엔진 색인 대상에서 제외
- 저장소 라이선스와 저작권 확인
- GitHub 연결 해제 시 처리 방식

### 3.7 게시물 및 프로젝트 콘텐츠 라이선스 정책

프로젝트 소개서, README, 스터디 기록, 자료실 파일의 권리 기준을 정한다.

포함 항목은 다음과 같다.

- 작성자는 원칙적으로 자신이 작성한 콘텐츠 권리를 가진다.
- Kobot 웹은 동아리 운영과 프로젝트 소개 목적 범위에서 표시할 수 있다.
- 외부 공개 프로젝트는 작성자가 외부 공개에 동의한 범위만 표시한다.
- 타인의 코드, 이미지, 논문, 자료를 올릴 때 출처와 라이선스를 확인한다.
- GitHub 저장소에 라이선스가 없으면 자유 사용 가능하다고 가정하지 않는다.

## 6. 탈퇴 후 산출물과 작성자 표시

탈퇴 후에도 프로젝트 소개서, 스터디 기록, 회의록, 팀 활동 기록, 자료 설명, 댓글 등 팀 산출물은 삭제하지 않는다.

다만 탈퇴자의 계정 개인정보는 삭제 또는 접근 제한하고, 산출물 작성자 표시는 최소화한다.

탈퇴 후 남기는 작성자 표시는 다음 범위로 제한한다.

- 닉네임
- 당시 공식 팀 소속
- 당시 프로젝트 역할
- 당시 조직 역할

닉네임과 팀/역할 정보도 다른 정보와 결합해 특정 개인을 알아볼 수 있으면 개인정보에 해당할 수 있다.

따라서 다음 조치를 적용한다.

- 회원가입 또는 콘텐츠 작성 시 탈퇴 후 산출물이 유지된다는 점을 고지한다.
- 개인정보 처리방침과 이용약관에 탈퇴 후 산출물 보존 목적과 표시 범위를 적는다.
- 멤버 워크스페이스 내부 표시와 외부 공개 페이지 표시를 구분한다.
- 멤버 워크스페이스 내부에서는 항상 닉네임으로 표시하고 익명 표시를 허용하지 않는다.
- 닉네임은 현재 활동 사용자 기준으로 중복을 허용하지 않는다.
- 최초 로그인 사용자는 프로필 설정 단계에서 닉네임을 반드시 입력해야 한다.
- 닉네임을 만들기 전에는 멤버 워크스페이스에 진입할 수 없다.
- 최초 프로필 설정에서 실명도 필수로 입력한다.
- 실명은 일반 멤버에게 공개하지 않고, 회원 승인과 운영 확인을 위해 승인권자 이상만 확인할 수 있다.
- 학번은 모든 가입자에게 필수로 받는다.
- 학번은 일반 멤버에게 공개하지 않고, 회원 승인과 운영 확인을 위해 승인권자 이상만 확인할 수 있다.
- 전화번호도 모든 가입자에게 필수로 받는다.
- 1차 구현에서는 전화번호 인증을 하지 않고 입력값만 받는다.
- 전화번호는 일반 멤버에게 공개하지 않고, 회원 승인, 운영 연락, 긴급 연락, 프로젝트 내부 소통, 장비 대여 또는 행사 운영처럼 필요한 경우에 사용한다.
- 전화번호는 필요한 경우에만 승인권자 이상이 확인할 수 있다.
- 전화번호 열람은 공식 팀, 프로젝트 팀, 스터디, 행사 운영 그룹, 장비 대여 또는 제작 작업 그룹처럼 같은 활동 단위에 함께 포함된 사람끼리 허용한다.
- 같은 활동 단위에 포함되어 있지 않은 일반 멤버는 해당 사용자의 전화번호를 볼 수 없다.
- 전화번호를 보는 행위 자체는 별도 감사 기록으로 남기지 않는다.
- 각 활동 단위의 참여자 목록에는 역할, 이름과 닉네임, 연락처, 학번, 학과, 소속 단과대, 소속 공식 팀을 표시할 수 있다.
- 내부 활동 단위의 참여자 목록에서는 `홍길동 (코봇러)`처럼 실명을 우선 표시하고 닉네임을 보조 표시한다.
- 공개 페이지에서는 실명을 기본 표시하지 않고, 사용자가 선택한 이름 공개, 닉네임 공개, 익명 표시 설정을 따른다.
- 참여자 목록은 해당 활동 단위에 포함된 구성원과 운영진만 볼 수 있다.
- 공개형 프로필 또는 기술 블로그형 페이지에는 닉네임, 기술 태그, 자기소개, 공개로 설정한 프로젝트 또는 글 정도만 표시한다.
- 공개형 프로필에서는 실명, 이메일, 전화번호, 학번을 기본 표시하지 않는다.
- 공개형 프로필에는 이메일 대신 연락 요청 기능을 제공한다.
- 1차 구현은 실시간 채팅이 아니라 연락 요청 보내기 형태로 시작한다.
- 모든 사용자는 다른 사용자에게 연락 요청을 보낼 수 있다.
- 연락 요청에는 연락 이유와 요청자가 연락 가능한 연락처를 첨부하게 한다.
- 연락 이유에는 최소 글자 수 제한을 두지 않는다.
- 연락 요청에 첨부 가능한 연락처 종류는 전화번호, 이메일, 카카오톡 ID, 디스코드 ID, GitHub, 기타로 둔다.
- 연락 요청을 보내려면 연락 가능한 연락처를 최소 1개 이상 첨부해야 한다.
- 연락 요청 전송에는 기본 일일 제한을 두지 않는다.
- 연락 요청을 받은 사용자는 해당 요청을 스팸성 연락으로 신고할 수 있다.
- 스팸성 연락 신고는 회장과 부회장이 확인하고 필요한 조치를 할 수 있다.
- 단순히 이유가 부족하거나 맥락 없이 연락한 경우에는 경고 조치를 기본으로 한다.
- 내용 없이 연락 요청을 반복한 경우에는 연락 기능 제한을 기본으로 한다.
- 욕설, 혐오, 성적 표현, 협박, 사칭, 악성 링크 등 이상한 내용으로 연락한 경우에는 계정 정지를 기본으로 한다.
- 스크래핑, 자동화, 봇, 매크로처럼 비정상적으로 대량 연락 요청을 보내는 경우에는 시스템이 먼저 차단한다.
- 자동 차단 대상은 회장과 부회장이 확인할 수 있다.
- 자동화 차단은 팝업 경고, 서버 측 탐지, 쿨다운, 운영진 검토 큐를 함께 사용한다.
- 10분 안에 연락 요청 3건 이상을 보내면 경고 팝업을 표시하고, 확인 시 해당 요청은 전송할 수 있다.
- 10분 안에 연락 요청 6건 이상을 보내면 10분 쿨다운을 적용한다.
- 1시간 안에 연락 요청 15건 이상을 보내면 회장과 부회장 검토 큐에 자동 등록한다.
- 동일하거나 유사한 내용을 여러 사용자에게 반복 전송하면 경고 후 반복 시 쿨다운을 적용한다.
- 같은 프로젝트나 같은 활동 단위 안에서 필요한 연락을 보내는 경우에는 탐지 기준을 완화할 수 있다.
- 봇성 패턴이 명확한 경우에는 즉시 연락 요청 기능을 제한하고 회장과 부회장 검토 대상으로 등록한다.
- 연락 요청은 로그인한 사용자만 사용할 수 있고, 상대방 이메일과 전화번호는 노출하지 않는다.
- 대시보드 사이드바에 `연락` 페이지를 두고 받은 연락 요청과 보낸 연락 요청을 확인하게 한다.
- 새 연락 요청이 도착하면 대시보드 알림에 표시한다.
- 연락 요청 상태는 `대기`, `수락`, `거절`로 구분한다.
- 연락 요청은 최대 3일 동안 대기 상태로 유지되고, 3일 동안 수락 또는 거절하지 않으면 자동으로 거절 처리한다.
- 자동 거절 사유는 `미응답`으로 기록하고 요청자에게 표시한다.
- 자동 거절된 요청은 다시 보낼 수 있고, 같은 사용자에게 다시 보내려면 24시간이 지나야 한다.
- 명시적으로 거절된 요청도 다시 보낼 수 있고, 같은 사용자에게 다시 보내려면 7일이 지나야 한다.
- 연락 요청이 수락되면 요청자가 첨부한 연락처가 수락자에게 보이고, 수락자의 연락처도 요청자에게 공개된다.
- 연락 요청을 수락할 때 수락자는 요청자에게 공개할 연락처를 선택한다.
- 수락자는 전화번호, 이메일, 카카오톡 ID, 디스코드 ID, GitHub, 기타 중 하나 이상을 선택하거나 직접 입력할 수 있다.
- 연락 요청을 수락하려면 수락자가 공개할 연락처를 최소 1개 이상 선택해야 한다.
- 연락 요청을 수락한 뒤에는 해당 요청에 대한 연락처 공개를 취소할 수 없다.
- 이미 수락된 연락 요청이 있어도 같은 사용자에게 다시 연락 요청을 보낼 수 있고, 수락 이력만을 이유로 재요청 제한을 두지 않는다.
- 연락 요청이 거절되면 수락자의 연락처는 공개하지 않는다.
- 연락 요청을 거절할 때 거절 사유를 선택사항으로 입력할 수 있다.
- 입력된 거절 사유는 요청자에게 표시한다.
- 연락 요청 내용과 첨부 연락처의 기본 보관 기간은 1년이다.
- 연락 요청을 받은 사용자는 받은 연락 요청을 삭제할 수 있다.
- 연락 요청을 보낸 사용자도 보낸 연락 요청을 삭제할 수 있다.
- 삭제는 각자 자신의 연락 페이지에서만 보이지 않게 처리하고, 한쪽 사용자가 삭제해도 상대방의 연락 페이지에서는 유지된다.
- 시스템 원본은 1년 보관 후 삭제한다.
- 연락 요청은 상세 내용과 첨부 연락처를 감사 기록에 복사하지 않고, 연락 요청 생성, 수락, 거절, 신고, 자동 차단 같은 이벤트만 감사 기록에 남긴다.
- 닉네임 중복 검사는 대소문자를 구분하지 않는다.
- 탈퇴한 사용자의 닉네임과 이전 닉네임은 재사용할 수 있다.
- 사용자 식별은 닉네임이 아니라 내부 사용자 ID를 기준으로 한다.
- 산출물과 기록에는 작성자 사용자 ID와 작성 당시 닉네임 스냅샷을 함께 저장한다.
- 닉네임 변경은 7일에 1번만 허용한다.
- 닉네임 변경은 금칙어와 부적절한 닉네임 검사를 통과해야 적용한다.
- 닉네임 길이는 2자 이상 12자 이하로 제한한다.
- 닉네임 입력에는 한글, 영문, 숫자, 공백만 허용한다.
- 닉네임 저장 시 공백은 `_`로 변환하고, 사용자가 직접 `_`를 입력하는 것은 허용하지 않는다.
- 닉네임 화면 표시 시 저장된 `_`는 공백처럼 표시한다.
- 닉네임이 변경되어도 기존 산출물과 기록의 작성자 표시는 작성 당시 닉네임으로 유지한다.
- 작성자 프로필을 열면 현재 닉네임과 `alpha - Beta - gamma(현)` 형태의 닉네임 변경 이력을 볼 수 있게 한다.
- 일반 멤버 프로필에는 닉네임 변경 이력을 최근 5개까지만 표시한다.
- 전체 닉네임 변경 이력은 회장과 부회장이 감사 기록에서 확인할 수 있다.
- 닉네임 변경 시 이전 닉네임을 일반 멤버에게 비공개 처리할지 사용자에게 묻는다.
- 이전 닉네임 공개 여부는 프로필 설정에서 언제든 다시 변경할 수 있다.
- 이전 닉네임 비공개를 선택해도 회장과 부회장은 감사와 운영 목적으로 확인할 수 있다.
- 감사 기록의 무결성을 위해 닉네임 변경 이벤트에는 이전 닉네임과 새 닉네임을 함께 남긴다.
- 공개 페이지는 프로젝트 중심으로 보여주고, 개인 표시 여부는 프로필 설정에서 관리한다.
- 공개 페이지 개인 표시 선택지는 이름 공개, 닉네임 공개, 익명 표시로 둔다.
- 공개 페이지 개인 표시 기본값은 익명 표시로 둔다.
- 탈퇴 요청 시 공개 페이지의 기존 산출물 작성자 표시 방식을 다시 선택하게 한다.
- 탈퇴자가 별도로 선택하지 않으면 공개 페이지에서는 익명 표시를 기본값으로 한다.
- 실명, 이메일, 학번, Google 프로필, 아이디는 탈퇴 후 산출물 표시에서 제거한다.
- 탈퇴자가 특정 산출물에 개인정보가 포함되어 있다고 요청하면 운영진이 검토해 작성자 표시 제거, 비공개 전환, 내용 수정, 삭제 중 필요한 조치를 한다.
- 공개 페이지에는 탈퇴자의 과거 팀과 역할을 기본적으로 표시하지 않는다.

## 7. 내부 운영 체크리스트

출시 전에 운영진이 확인해야 할 항목은 다음과 같다.

- 실제 수집 항목과 DB 컬럼이 개인정보 처리방침과 일치하는지 확인한다.
- Supabase 프로젝트는 `Northeast Asia (Seoul)`, `ap-northeast-2` 리전으로 생성하고 데이터 처리 조건을 확인한다.
- Google OAuth 동의 화면의 앱 이름은 `KOBOT`으로 설정하고, 도메인과 개인정보 처리방침 URL을 설정한다.
- Google OAuth 사용자 유형은 `External`로 설정한다.
- 공식 웹 도메인은 `kobot.kookmin.ac.kr`로 설정한다.
- 개인정보 처리방침 URL은 `https://kobot.kookmin.ac.kr/privacy`, 이용약관 URL은 `https://kobot.kookmin.ac.kr/terms`로 설정한다.
- Resend 발신 주소로 `@kookmin.ac.kr` 또는 `@kobot.kookmin.ac.kr`을 사용하려면 학교의 이메일 발신 인증 DNS 레코드 설정 협조가 필요하다.
- 학교 DNS 협조가 어렵다면 별도 발신 도메인을 준비하거나, 학교 공식 메일은 Reply-To 또는 문의처로만 사용하는 방식을 검토한다.
- GitHub App 권한 범위가 README 조회에 필요한 최소 권한인지 확인한다.
- private repository README를 프론트엔드에 장기 캐시하지 않는다.
- 회원 탈퇴 요청 승인 후 어떤 데이터가 삭제되고 어떤 데이터가 1년 보존되는지 정한다.
- 1년이 지난 감사 기록은 기본적으로 삭제되도록 구현한다.
- 개인정보 문의 이메일은 `kobot@kookmin.ac.kr`로 둔다.
- 운영 문의 이메일은 `kobot@kookmin.ac.kr`로 둔다.
- 회장 유고 시 최고 관리자 복구는 부회장 1명과 공식 팀장 2명 승인 조합으로 처리한다.
- 정식 회장 임명 투표권자는 코봇 활동 부원 전체이며, 외부 프로젝트 참여자 또는 승인 대기 사용자는 투표권을 가지지 않는다.
- 투표 기록의 기본 보관 기간은 1년이며, 보관 기간이 지난 투표 기록은 삭제한다.
- 개별 투표 기록은 1년 후 삭제하지만, 개별 투표자의 선택 내역을 포함하지 않는 투표 결과 요약은 안건 기록 또는 회의록에 남길 수 있다.
- 투표 결과 요약에는 안건명, 투표 기간, 투표권자 범위, 총 투표수, 결과, 최종 상태, 공개된 범위의 후보별 또는 선택지별 득표수를 남길 수 있다.
- 정식 회장 후보는 코봇 활동 부원 누구나 될 수 있다.
- 임시 회장이 지정된 날부터 14일 동안 후보 참여 기간을 두고, 임시 회장 지정 후 15일째 정식 회장 임명 투표를 연다.
- 정식 회장 임명 투표 기간은 3일이며, 임시 회장은 정식 회장 후보에 기본으로 자동 등록된다.
- 임시 회장은 정식 회장 후보에서 사퇴할 수 있고, 후보 사퇴도 기록으로 남긴다.
- 코봇 활동 부원은 직접 후보로 참여하거나 다른 코봇 활동 부원을 추천할 수 있으며, 추천을 받은 사용자는 알림을 받고 수락해야 후보가 된다.
- 정식 회장 임명 투표는 익명 투표로 진행하고, 누가 투표했는지는 중복 투표 방지를 위해 기록할 수 있지만 누가 어떤 후보에게 투표했는지는 운영진도 확인할 수 없게 한다.
- 투표 결과는 투표 종료 후 공개하고, 진행 중에는 중간 득표율과 후보별 득표 수를 공개하지 않는다.
- 정식 회장 임명 투표에는 별도 투표율 기준을 두지 않으며, 당선자는 최다 득표자로 정한다.
- 최다 득표자가 2명 이상이면 동률 후보만 대상으로 결선 투표를 진행한다.
- 결선 투표 기간도 3일이다.
- 결선 투표에서도 동률이면 부회장 1명과 공식 팀장 2명의 승인 조합으로 최종 당선자를 결정하고, 최종 결정 과정과 승인자는 감사 기록으로 남긴다.
- 투표 알림은 대시보드 알림, 사이드바 배지, 이메일 알림으로 보내며, 투표 시작, 마감 24시간 전, 마감 3시간 전, 투표 종료, 결과 공개 시 알림을 보낸다.
- 1차 전체 알림 채널은 웹 알림, 사이드바 배지, 이메일 알림으로 두고, 카카오톡 알림과 문자 알림은 1차 범위에서 제외한다.
- 이메일 발송 서비스는 1차 구현에서 Resend를 사용한다.
- Resend 발신 주소는 학교 DNS 협조 가능 여부에 따라 확정한다.
- 운영 문의와 개인정보 문의 공식 메일은 `kobot@kookmin.ac.kr`을 사용한다.
- 이메일 발신 도메인 연동은 1차 개발 완료 후 학교 담당자에게 DNS 레코드 등록 요청 메일을 보내 진행한다.
- 이메일 알림은 중요 알림과 일반 알림으로 나누고, 중요 알림은 기본 켜짐, 일반 알림은 사용자가 선택해서 켤 수 있게 한다.
- 사용자는 중요 이메일 알림도 끌 수 있지만, 끌 때 중요한 승인, 투표, 권한 요청을 놓칠 수 있다는 경고 문구를 표시한다.
- 웹 알림은 읽음과 안읽음 상태를 가지고, 사용자는 개별 알림 또는 전체 알림을 읽음 처리할 수 있다.
- 사용자는 개별 웹 알림 또는 전체 웹 알림을 삭제할 수 있고, 삭제는 사용자 화면에서만 보이지 않게 처리한다.
- 웹 알림 삭제가 감사 기록이나 관련 원본 기록을 삭제하지는 않는다.
- 웹 알림의 기본 보관 기간은 1년이며, 보관 기간이 지난 웹 알림은 삭제한다.
- 투표 후 당선자가 확정되면 당선자에게 회장 권한 수락 요청을 보내고, 당선자가 수락하면 회장 권한을 반영한다.
- 당선자가 3일 안에 수락하지 않으면 당선 포기로 처리한다.
- 당선자가 명시적으로 거절하면 즉시 당선 포기로 처리한다.
- 당선 포기 시 차순위 후보에게 바로 권한 수락 요청을 보내지 않고 정식 회장 임명 투표를 다시 진행한다.
- 재투표에서도 결론이 나지 않거나 출마자가 없으면 출마자가 나올 때까지 부회장이 회장 권한을 대신 수행한다.
- 정식 회장 후보가 단일 후보인 경우 찬성/반대 투표로 진행한다.
- 단일 후보 찬반 투표에서는 찬성이 반대보다 많으면 통과한다.
- 단일 후보 찬반 투표에서 반대가 찬성보다 많으면 후보 모집부터 다시 시작한다.
- 단일 후보 찬반 투표에서 찬성과 반대가 같으면 부회장 1명과 공식 팀장 2명의 승인 조합으로 최종 결정한다.
- 당선자 수락과 권한 반영은 감사 기록으로 남긴다.
- 투표 서비스는 회장 선출 외 일반 안건 투표에도 사용할 수 있다.
- 일반 안건 투표는 코봇 활동 부원 전체, 특정 공식 팀, 특정 프로젝트 팀, 특정 스터디, 특정 행사 또는 활동 단위 참여자처럼 투표권자 범위를 선택할 수 있다.
- 일반 안건 투표는 회장, 부회장, 공식 팀장, 프로젝트 팀장, 스터디장이 자기 권한 범위 안에서 만들 수 있다.
- 일반 안건 투표는 생성자가 익명 투표 또는 기명 투표를 선택할 수 있다.
- 일반 안건 투표의 기본값은 익명 투표다.
- 일반 안건 투표 결과 공개 시점은 생성자가 투표 진행 중 공개 또는 투표 종료 후 공개 중 선택할 수 있다.
- 일반 안건 투표 결과 공개 시점의 기본값은 투표 종료 후 공개다.
- 투표 생성 시 즉시 오픈 또는 예약 오픈을 선택할 수 있고, 예약 오픈은 지정한 시간에 자동으로 시작된다.
- 일반 안건 투표의 종료 시간은 생성자가 직접 설정한다.
- 일반 안건 투표의 최대 기간은 30일이다.
- 1차 일반 안건 투표 유형은 찬반 투표, 단일 선택 투표, 복수 선택 투표를 지원하고, 서술형 투표는 이후 확장으로 둔다.
- 복수 선택 투표에서는 생성자가 최대 선택 개수를 설정할 수 있고, 기본값은 최대 선택 개수 제한 없음이다.
- 일반 안건 투표는 오픈 전에는 문항, 선택지, 투표 설정을 수정할 수 있고, 투표 시작 후에는 문항과 선택지를 수정할 수 없다.
- 투표 시작 후 오타 또는 내용 오류가 있으면 기존 투표를 취소하고 새 투표를 생성한다.
- 투표 취소는 생성자와 생성자의 상위 권한자가 할 수 있고, 투표 취소는 감사 기록으로 남긴다.
- 이용약관과 개인정보 처리방침 변경 이력을 남긴다.
